## Safe Compact Diagnostic Contract

Future user-facing/readiness diagnostics may expose only compact, stable, no-leak fields:

- module: the affected module, currently only `tmuxSnapshotParse`.
- capability: the requested parser capability, such as tmux snapshot parsing.
- status/resultMarker: compact status/result marker such as `unknown`, `stale`, `unavailable`, `unsupported`, or `failed`.
- failure kind: stable failure kind from the documented fail-closed vocabulary.
- short remediation: one short user-safe remediation sentence, such as reinstall extension, use TypeScript mode, disable preview, or roll back to a known-good version.
- supported-platform/freshness hint: compact hint that the current platform is unsupported, helper package is absent, helper version is stale, or packaged artifact is not fresh enough.
- release decision pointer: pointer to this readiness gate or a later explicit release decision doc, not raw package internals.

The diagnostic contract must stay compact. It must not become a transport for helper logs, state files, package internals, mailbox content, or worker prompt text.

## Forbidden Fields and Leaks

Future user-facing/readiness diagnostics must not expose:

- helper path.
- helper stdout/stderr.
- repository path, repo path, or cwd path.
- raw `cutoverReason`.
- raw state JSON or raw team JSON.
- sidecar/cache/index contents.
- raw manifests/checksums/provenance payloads.
- worker prompts.
- stack traces.
- mailbox/report text.
- package internals.
- hidden runtime state, environment variable bodies, task report bodies, or full-text message content.

If a later implementation needs one of these values for internal debugging, it must stay behind developer-only logs or tests and must not reach `/team`, readiness summaries, public command output, task summaries, or report excerpts.

## Failure Policy Expectations

Future cutover/preview failure paths must preserve current safety expectations:

- no pane loss.
- no false successful empty snapshot.
- no stale success when parser ownership is cutover/preview and the helper failed.
- no force reconcile.
- no kill panes.
- no worker lifecycle mutation.
- no task/report governance mutation.
- no hidden TypeScript parser fallback in cutover/preview failure paths.
- no fallback counter or metadata that implies a hidden successful TypeScript parser fallback for `tmuxSnapshotParse` in `go-cutover` or `go-packaged-preview` failure paths.

The expected failure posture is fail closed with a compact diagnostic. It is not fail open, silent recovery, destructive reconciliation, or lifecycle mutation.

## Release Decision Gate

STOP future real native package metadata, native artifact inclusion, npm publication, default/native cutover, resolver defaulting, or fallback removal until explicit user approval after all of these are proven:

- compact diagnostics UX is implemented and no-leak guarded.
- generated native helper artifacts exist for the supported OS/arch/libc matrix.
- generated manifests, checksums, provenance, license metadata, and executable-bit validation are available and guarded.
- clean install smokes pass across supported platforms without a Go toolchain, source checkout, manual helper env, lifecycle download, install-time build, or hidden network fetch.
- unsupported-platform behavior and remediation text are accepted for every AgentTeam-supported platform.
- current `go-cutover` and explicit `go-packaged-preview` failure policies prove no pane loss, no false empty snapshots, no force reconcile, no kill panes, no worker lifecycle mutation, and no hidden TypeScript parser fallback.
- rollback story covers bad package metadata, bad helper artifact, checksum/provenance mismatch, unsupported platform, and package unpublish/deprecation scenarios.
- package release owner accepts package naming, versioning, provenance, attestation, checksum, license, supported-platform matrix, and rollback responsibilities.
- TypeScript parser fallback deletion remains blocked until native/default availability and diagnostics are separately proven.
- `compactReadModelFingerprint` remains non-cutover unless a later approved slice solves its own availability and fallback policy.
- Go helper authority remains parser-only/stdin-stdout `tmuxSnapshotParse` and does not gain tmux execution/capture, state/repository writes, network/listeners, worker lifecycle, PlanRun/governance, full-text access, package/release authority, or control-plane behavior.

## Slice 2 Compact Diagnostics Model Evidence

Slice 2 adds an internal, pure/read-only compact diagnostics model for `tmuxSnapshotParse` parser unavailability. The model is testable through `core/kernelDiagnostics.ts` and guarded by `tests/suites/go-kernel-v0423-compact-diagnostics-model.cjs`.

The model maps every current fail-closed cutover failure kind to the Slice 1 safe fields only:

- `missing-helper`: unavailable helper summary with reinstall, TypeScript mode, or packaged preview disablement remediation and platform hint.
- `disabled-helper`: disabled-before-ownership summary with TypeScript mode or explicit-prerequisite remediation and freshness hint.
- `helper-unsupported-protocol`: protocol mismatch summary with update-together or rollback remediation and freshness hint.
- `helper-unsupported-version`: version skew summary with update-together or rollback remediation and freshness hint.
- `helper-unsupported-capability`: missing `tmuxSnapshotParse` capability summary with TypeScript mode remediation and freshness hint.
- `helper-timeout`: timeout summary with TypeScript mode or rollback remediation and freshness hint.
- `helper-spawn-error`: startup failure summary with TypeScript mode plus reinstall/rollback remediation and platform hint.
- `helper-crash`: crash summary with TypeScript mode plus reinstall/rollback remediation and freshness hint.
- `helper-nonzero-exit`: failed process result summary with TypeScript mode plus reinstall/rollback remediation and freshness hint.
- `helper-empty-response`: empty payload summary with TypeScript mode plus helper-health remediation and freshness hint.
- `helper-malformed-json`: malformed protocol payload summary with TypeScript mode plus update/rollback remediation and freshness hint.
- `helper-jsonrpc-error`: compact protocol error summary with TypeScript mode plus health/capability remediation and freshness hint.
- `helper-incompatible-response`: parser contract mismatch summary with TypeScript mode plus compatible-helper remediation and freshness hint.
- `helper-unsafe-response-shape`: unsafe parser output summary with TypeScript mode plus update/rollback remediation and freshness hint.
- `previous-helper-failure`: prior fail-closed disablement summary with TypeScript mode or restart-after-resolution remediation and freshness hint.

Slice 2 implementation constraints:

- diagnostics are internal/testable only and are not rendered in `/team` or public runtime UI surfaces.
- diagnostics expose only `module`, `capability`, `status`, `resultMarker`, `failureKind`, short `remediation`, optional `platformHint`, optional `freshnessHint`, and release decision pointer.
- diagnostics do not include helper path, stdout/stderr, repo/cwd, raw `cutoverReason`, raw state/team JSON, sidecar/cache/index contents, raw manifests/checksums/provenance payloads, worker prompts, stack traces, mailbox/report text, or package internals.
- the helper performs no filesystem, state, tmux, process, network, environment, package, or lifecycle access.
- default/unset remains disabled/TypeScript; `go-packaged-preview` remains explicit-only; current `go-cutover` remains unchanged; `compactReadModelFingerprint` remains non-cutover; runtime `/team` remains quiet.
- STOP for runtime UI diagnostics, native artifact approval, real package metadata, default resolver changes, `go-packaged-preview` availability semantics changes, `go-cutover` behavior changes, TypeScript parser fallback deletion, or broader Go authority.
