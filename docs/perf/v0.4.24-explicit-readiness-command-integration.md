## Contract Boundary

The future command/readiness surface is explicit opt-in and reviewer-facing. It may be used by a later approved command or test harness to inspect compact `tmuxSnapshotParse` cutover/packaged-preview readiness diagnostics.

Required boundary:

- explicit invocation only; no ambient `/team` panel rendering.
- reviewer-facing readiness summary only; not normal-user native availability proof.
- read-only and side-effect-free.
- derived from v0.4.23 compact diagnostics/readiness helpers, especially `summarizeTmuxSnapshotParseFailureDiagnostic()` and `formatTmuxSnapshotParseFailureReadiness()` in `core/kernelDiagnostics.ts`.
- scoped to parser-only stdin/stdout `tmuxSnapshotParse` diagnostics; no broader Go helper authority.

## Allowed Command/Readiness Output

A later command/readiness implementation may output only compact safe fields/text from the v0.4.23 helpers:

- module.
- capability.
- status.
- resultMarker.
- failureKind.
- remediation.
- hint derived only from safe `platformHint` or `freshnessHint`.
- releaseDecision pointer.
- compact summary text that is deterministically derived from module, status/resultMarker, and failureKind.

Allowed output must stay compact, stable, and safe for reviewer-facing command output. It must not expand into logs, raw payloads, state dumps, mailbox text, package internals, or helper execution details.

## Forbidden Fields and Leaks

A later command/readiness implementation must not expose:

- helper path.
- helper stdout/stderr.
- repository path, repo path, or cwd path.
- raw `cutoverReason`.
- raw state JSON or raw team JSON.
- sidecar/cache/index contents.
- raw manifests/checksums/provenance payloads.
- worker prompts.
- stack traces.
- mailbox/report text.
- package internals.
- env bodies or environment variable values.
- full-text content from messages, reports, prompts, task bodies, state files, manifests, checksums, provenance payloads, helper logs, stdout, or stderr.

Forbidden values must not reach command output, readiness summaries, `/team`, task summaries, report excerpts, or public runtime surfaces.

## Read-Only Behavior Contract

A later explicit readiness command must remain read-only:

- no state writes.
- no mailbox/report full-text reads.
- no task/report governance mutation.
- no tmux execution/capture beyond existing product paths.
- no worker lifecycle mutation.
- no pane reconcile.
- no kill panes.
- no force reconcile.
- no helper download, build, install, package, version, publish, or lifecycle behavior.
- no network clients, servers, listeners, or hidden fetches.

The command/readiness surface is a formatter/summary boundary, not a control-plane operation.

## Slice 2 Command Surface Discovery and Seam Selection

Slice 2 audits existing command/tool surfaces and selects the smallest future integration seam for Slice 3. This slice is docs/tests only and no command is implemented in Slice 2.

Audited surfaces:

- `api/commands.ts`: registers AgentTeam commands through `registerAgentTeamCommands()` and currently delegates only to `registerTeamCommands()`.
- `commands/team.ts`: owns the single public `/team` command, routes `config` subcommands through `handleTeamConfigCommand(args, ctx)`, and otherwise opens the `/team` panel.
- `commands/config.ts`: provides the existing explicit subcommand parser/formatter pattern for `/team config init|show|validate|migrate --dry-run`; `show`, `validate`, and `migrate --dry-run` demonstrate command text output and read-only/dry-run conventions, while `init` is explicitly writeful and therefore not the readiness model.
- `commands/shared.ts`: defines command dependencies; future readiness should avoid adding state/tmux/write dependencies unless separately approved.
- `api/tools.ts`: registers tool surfaces through `registerAgentTeamTools()`; tools are model-callable workflow operations and are not the smallest reviewer-facing command seam.
- `tools/`: contains create/spawn/message/task/report/plan-run tool handlers with mailbox, state, task/report governance, outbox, and worker delivery behavior; these are too broad for a read-only readiness command.
- `tests/suites/commands.cjs`: verifies only `team` is registered and config subcommands are routed through notification text; future Slice 3 should add similarly focused command tests without expanding public command count unexpectedly.
- `tests/suites/public-output-leak-guards.cjs`: guards public output vocabulary/no-leak boundaries and should remain a reference for compact readiness text.
- `tests/suites/go-kernel-v0424-readiness-command-contract-docs.cjs`: freezes the Slice 1 command/readiness contract before implementation.

Recommended smallest Slice 3 seam:

- add a new explicit `/team readiness` subcommand handled before `openTeamPanel()`, using the same subcommand-dispatch style as `/team config`.
- keep the seam inside command handling, likely `commands/team.ts` routing to a small future helper such as `handleTeamReadinessCommand(args, ctx)` or equivalent.
- have the future helper call only the existing safe v0.4.23 readiness helpers from `core/kernelDiagnostics.ts`, such as `summarizeTmuxSnapshotParseFailureDiagnostic()` and `formatTmuxSnapshotParseFailureReadiness()`.
- emit compact reviewer-facing notification text through the command response path, not through `/team` panel rendering.
- keep the future command explicit opt-in, read-only, deterministic, and testable with command-style notification assertions.

Why not other seams:

- Do not add a new model-callable tool in Slice 3 unless separately approved; tools have broader workflow authority and are not the minimal reviewer-facing command surface.
- Do not render readiness diagnostics in `/team`; ambient panel rendering would violate the quiet `/team` invariant.
- Do not add package/native resolver behavior; the readiness command is not normal-user native availability proof.
- no tmux capture, state writes, mailbox/report full-text reads, task/report mutation, worker lifecycle behavior, pane reconcile, or pane kill behavior.

Slice 3 may do only this, after separate approval:

- implement an explicit `/team readiness` command or equivalent subcommand seam.
- format compact readiness output using only allowed fields from the v0.4.23 helper contract: module, capability, status, resultMarker, failureKind, remediation, hint, releaseDecision.
- add command tests that verify explicit invocation, stable compact output, no forbidden leaks, no `/team` rendering, no state writes, no full-text reads, no task/report governance mutation, no tmux execution/capture authority beyond existing product paths, no worker lifecycle mutation, no pane reconcile, and no kill panes.

Slice 3 must not do this:

- must not implement `/team` ambient UI or runtime panel diagnostics rendering.
- must not add a new model-callable tool without a later explicit owner decision.
- must not change runtime behavior, default Go, current `go-cutover`, or `go-packaged-preview` availability semantics.
- must not delete the TypeScript parser fallback.
- must not broaden Go authority beyond parser-only stdin/stdout `tmuxSnapshotParse`.
- must not change package/native behavior, `package.json`, `optionalDependencies`, lifecycle hooks, helper downloads, lockfiles, Go modules, native artifacts, npm version, or npm publish.

The Slice 1 allowed output, forbidden leak, read-only behavior, runtime invariant, and package/native STOP gate sections remain authoritative for the future Slice 3 seam.

Slice 3 implementation note: `/team readiness` is now the explicit command seam selected here. It remains opt-in, read-only, deterministic, and routed before `openTeamPanel()`.
