# v0.4.18 tmuxSnapshotParse Cutover Plan

> Scope: tmuxSnapshotParse-specific cutover plan for v0.4.18. This is docs/tests-only planning; it does not delete the TypeScript runtime fallback, change runtime behavior, package a native helper, or change npm/package release mechanics.

## Purpose

`tmuxSnapshotParse` is the first recommended Go-owned module candidate because it is a deterministic parser over stdout that TypeScript has already captured. It is lower risk than read-model fingerprinting or write-side paths: it does not need repository state, sidecars, outbox writes, task/report governance, PlanRun authority, mailbox/report full text, tmux subprocess ownership, or worker lifecycle control.

This plan specializes the module gate from `docs/decisions/0002-module-owned-go-kernel-cutover.md`, `docs/perf/v0.4.18-go-module-cutover-checklist.md`, and the fail-closed diagnostics contract in `docs/perf/v0.4.18-go-cutover-fail-closed-diagnostics.md`. It keeps v0.4.18 as planning/checklist work only; future implementation must pass the gate before any runtime fallback deletion.

## Go-Owned Boundary

After a future cutover, Go may own only this bounded capability:

- Parse TypeScript-captured tmux snapshot stdout using `tmuxSnapshotParse`.
- Return `TmuxSnapshot`-compatible compact data: `capturedAt`, ordered `panes`, `byPaneId`, `ok`, and explicit compact diagnostics.
- Preserve TypeScript-validated protocol/helper-version/capability expectations before accepting output.

TypeScript/pi remains responsible for:

- Running and capturing tmux commands such as `list-panes`.
- Pane creation, pane labels, pane/window lifecycle, light/force reconcile policy, and worker lifecycle.
- `/team` rendering, panel data-source authority, task/report governance, PlanRun, state/repository compatibility, full-text boundaries, and user-facing control flow.
- npm package metadata, package version, native packaging decisions, and release mechanics.

Go must not execute tmux, inspect live pane lifecycle, spawn or kill panes, send keys, mutate state, move governance/full-text boundaries, or become a package/release control surface.

## Readiness Criteria

A future cutover PR must show all of these PASS signals before `tmuxSnapshotParse` is treated as Go-owned:

- **Parity corpus:** `tests/fixtures/kernel/tmux/snapshotCases.cjs` and `tests/suites/go-kernel-tmux-snapshot-parser.cjs` cover TypeScript parser, direct Go helper, and adapter-backed helper output equivalence.
- **Helper health/protocol:** helper health returns protocol version `1`, helper version `0.3.0-read-model-shadow` or an explicitly reviewed successor, required `tmuxSnapshotParse` capability, and `businessPathsConnected:false`.
- **Failure coverage:** missing helper, disabled helper, protocol/helper-version mismatch, missing capability, malformed JSON, unsafe output shape, JSON-RPC error, timeout, crash/nonzero exit, and prior-helper-failure cases are covered.
- **Boundary scans:** Go helper source has no tmux execution, state/repository, sidecar/outbox, governance, full-text, worker lifecycle, network/listener, or package/release authority.
- **Package/native sanity:** no `package.json` version change, no `npm version`, no `npm publish`, no native packaging, no checked-in native binaries, no package lock files, no `go.mod`, and no `go.sum`.
- **Rollback text:** rollback uses prior GitHub tag/npm version or a corrected npm version, not a hidden TypeScript runtime fallback.
- **Reviewer signoff:** module owner and release/checklist reviewer explicitly sign off on parity, failure handling, fallback deletion targets, boundaries, and rollback.

## Parser Semantics To Preserve

The Go-owned parser must preserve the current TypeScript and v0.4.17 corpus semantics exactly:

- Empty stdout returns an empty successful snapshot.
- Trailing newline and CRLF input keep the same row interpretation; CRLF rows normalize a terminal `\r`.
- Malformed rows and rows with too few tab fields are skipped.
- Empty pane id rows are skipped.
- Empty label and empty current command fields are retained.
- Duplicate pane ids preserve first-seen `panes` order and last-seen field values.
- Extra tab fields after `currentCommand` are ignored.
- Unicode labels and commands are preserved.
- `panes` order and `byPaneId` contents stay consistent with the same pane items.
- Successful parses return `capturedAt`, `panes`, `byPaneId`, `ok:true`, and no parser error.

## Future Cutover Failure Handling

After a future cutover, parser unavailability is an unknown/stale snapshot condition, not proof that panes disappeared.

The tmux-specific failure policy for this module is:

- Missing, incompatible, timed-out, crashed, malformed, or unsafe Go parser output fails closed for `tmuxSnapshotParse` with compact diagnostics per `docs/perf/v0.4.18-go-cutover-fail-closed-diagnostics.md`.
- Parser failure must not be interpreted as pane loss and must not emit an empty successful pane list.
- Parser failure must not destructively update worker or pane state, clear pane bindings, kill panes, mark workers error, or trigger force reconcile.
- A light refresh must not mark workers error or force reconcile solely because the parser failed.
- Runtime UI may show an unknown/stale snapshot state only after a future user-facing diagnostics surface is separately designed; `docs/perf/v0.4.18-go-cutover-fail-closed-diagnostics.md` is the compact bench/docs/debug contract, not a runtime `/team` UI implementation.

## Future Deletion Targets

Do not delete these paths in v0.4.18 Slice E. They are future cutover targets only after the readiness gate passes:

- `tmux/snapshot.ts`: remove the runtime TypeScript parser fallback path for `parseTmuxPaneSnapshot` after Go owns `tmuxSnapshotParse`.
- `core/kernel.ts`: remove or retire adapter fallback behavior that silently re-enters the TypeScript parser for this capability after Go selection.
- Tests/docs that assert permanent parser fail-open fallback must be updated to distinguish pre-cutover migration fallback from post-cutover fail-closed unknown/stale handling.

Until that future cutover, the TypeScript parser remains the active compatibility baseline and no runtime behavior changes are made.

## Non-Goals

This plan does not approve or implement:

- deleting the TypeScript runtime parser fallback in this slice.
- Go tmux execution, tmux capture, pane lifecycle, worker lifecycle, light/force reconcile, or `/team` ownership.
- Go state writes, repository writes, sidecar/outbox writes, task/report governance, PlanRun authority, or full-text boundary movement.
- whole-product Go rewrite, default Go control plane, Go daemon, scheduler, worker process manager, or package/release control.
- package version changes, `npm version`, `npm publish`, native packaging, package lock files, `go.mod`, `go.sum`, checked-in native binaries, commit, tag, or push.
