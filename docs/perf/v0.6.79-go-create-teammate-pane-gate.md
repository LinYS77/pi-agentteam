# v0.6.79 Go Create Teammate Pane Gate

Result: v0.6.79 defines the high-risk `tmux/panes.ts createTeammatePane(...)` Go cutover gate without implementing runtime mutation.

This is gate-only evidence for exactly one future candidate: `workerLifecycle.createTeammatePane` backing private/public `tmux/panes.ts createTeammatePane(...)`.

No Go handler, TypeScript adapter method, `tmux/panes.ts` cutover, native helper rebuild, or package/release action is made in this slice.

## Current TypeScript-Owned Behavior

`createTeammatePane(input, signal)` remains TypeScript-owned and still starts by calling `ensureSwarmWindow(input.preferred, signal)`.

`ensureSwarmWindow(input.preferred, signal)` remains TypeScript-owned. Any current `new-session` or `new-window` behavior inside `tmux/windows.ts` is outside this gate and is not authorized for future `workerLifecycle.createTeammatePane`.

Inside `createTeammatePane(...)`, current pane discovery remains:

- `tmux list-panes -t <swarm.target> -F '#{pane_id}'`

The current first-pane leader-layout split remains:

- `tmux split-window -t <leaderPaneId> -h -p 34 [-c <cwd>] -P -F '#{pane_id}' [startCommand]`

The current later/other split remains:

- `tmux split-window -t <lastPaneId> -v [-c <cwd>] -P -F '#{pane_id}' [startCommand]`

The current layout commands remain:

- `tmux select-layout -t <swarm.target> main-vertical`
- `tmux select-layout -t <swarm.target> tiled`

The current leader resize command remains:

- `tmux resize-pane -t <leaderPaneId> -x 66%`

Post-create labeling and refresh remain outside this gate's new command authorization. Existing v0.6.76/v0.6.78 pane label helpers and v0.6.74 label refresh behavior remain unchanged; this gate does not re-authorize broad label orchestration.

## Authorized Future Go Command Surface

This gate authorizes only the future pane discovery, creation, and layout/resize command surface for a later `workerLifecycle.createTeammatePane` cutover:

- `tmux list-panes -t <swarm.target> -F '#{pane_id}'`
- `tmux split-window -t <leaderPaneId> -h -p 34 [-c <cwd>] -P -F '#{pane_id}' [startCommand]`

- `tmux select-layout -t <swarm.target> tiled`
- `tmux resize-pane -t <leaderPaneId> -x 66%`

No other Go mutating tmux commands are authorized by this gate.

Future Go must pass `cwd` and `startCommand` as argv values only, never shell-interpolate them.

`cwd` and `startCommand` are high-risk opaque user/runtime data. A future helper must validate them compactly, must not log raw values, and must not echo them through helper diagnostics, errors, task reports, or tests.

Future Go should validate pane ids and targets compactly before using them as tmux targets. Raw tmux stdout/stderr/helper output must not leak through public behavior or compact diagnostics.

## Future Cutover Rules

The future cutover is not implemented here. When a later task performs it:

- TypeScript/pi remains the public facade and pi extension compliance boundary.
- `ensureSwarmWindow(...)` remains TypeScript-owned unless a separate explicit gate authorizes session/window creation ownership.
- future `workerLifecycle.createTeammatePane` may own only the pane discovery, `split-window`, `select-layout`, and `resize-pane` surface listed above.
- The future operation should preserve current thrown create/layout failure behavior, because current `createTeammatePane(...)` uses throwing `runTmuxAsync(...)` for pane creation/layout commands.
- future failures should fail closed with compact internal diagnostics.
- raw stdout/stderr/helper output and raw `cwd`/`startCommand` text must not leak through diagnostics or reports.
- after a future cutover, `tmux/panes.ts createTeammatePane(...)` must not retain hidden TypeScript fallback for the same pane discovery/creation/layout behavior.
