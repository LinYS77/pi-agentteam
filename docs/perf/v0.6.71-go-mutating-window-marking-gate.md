# v0.6.71 Go Mutating Window Marking Gate

Result: v0.6.71 defines the first explicit mutating tmux Go cutover gate without implementing runtime mutation.

This is a contract/evidence slice only. Current runtime behavior remains unchanged from v0.6.70, and the accepted-but-uncommitted v0.6.70 `findWindowTargetByName` cutover remains the latest actual Go runtime migration.

## Current State

Current state: Go source still has no mutating tmux commands.

The current Go `workerLifecycle` command surface remains read-only or existence/availability oriented: pane/window/session inspection, current binding capture, and tmux availability. It must not add `set-option`, `new-session`, `new-window`, `split-window`, `select-layout`, `resize-pane`, `send-keys`, `kill-pane`, pane label/title mutation, state writes, task/report governance, UI view-model ownership, or release/package verification in this gate.

`tmux/labels.ts markWindowAsAgentTeam(target, signal)` remains TypeScript-owned in v0.6.71 and still performs the existing no-throw window marking calls after the explicit `windowExists(target, signal)` guard.

## Authorized Next Runtime Candidate

Authorized next runtime candidate: `markWindowAsAgentTeam(target, signal)` only.

A future implementation slice may move only this non-destructive window-marking behavior behind a Go `workerLifecycle` operation. That future slice may use only these exact tmux command surfaces:

- `tmux set-option -w -t <target> automatic-rename off`
- `tmux set-option -w -t <target> allow-rename off`
- `tmux set-option -w -t <target> @agentteam-window 1`

No other Go mutating tmux commands are authorized by this gate.

The authorization is intentionally narrow: it covers only window-level marking needed to identify the AgentTeam window and to prevent tmux automatic renaming from fighting the facade. It does not authorize `refreshWindowPaneLabels(target, signal)` pane border formatting, pane labels, pane titles, pane creation, lifecycle, wake, kill, state, task, UI, or release/package migration.

## Future Facade Rule

After the actual future cutover, `tmux/labels.ts markWindowAsAgentTeam` must not retain direct TypeScript `runTmuxNoThrowAsync(['set-option', '-w'...])` fallback for the same behavior.

The facade rule is the same cutover rule used by the read-only slices: once a facade is cut to Go, the TypeScript facade remains the public/pi compliance boundary, but it must not keep a hidden runtime implementation for the same behavior. Rollback should be by reverting the explicit slice or version, not by keeping two active implementations.

The future adapter call, if implemented, must be an explicit TypeScript-governed seam. Go must not autonomously mark arbitrary windows, discover/create sessions, create windows, or perform broader lifecycle orchestration.

## Public Behavior Contract For Future Slice

The public facade remains no-throw/void.

The current TypeScript `markWindowAsAgentTeam(target, signal)` behavior uses no-throw tmux calls and returns `Promise<void>`. The future Go-backed facade must preserve that public contract:

- success resolves `void`.
- helper failure, invalid target, and abort should produce compact diagnostics internally and must not leak raw stdout/stderr/helper output through the public facade.
- invalid or missing target must fail closed without creating sessions/windows.
- cancellation/abort must remain compact and no-throw at this facade unless a future task explicitly changes the public contract.

## Window Existence Authority

Window existence authority remains explicit: keep the TypeScript `windowExists(target, signal)` guard or move it only inside the same future slice if documented and tested.

This gate does not authorize broad window/session creation. If the future slice moves the existence check into Go, that check must be part of the same `markWindowAsAgentTeam(target, signal)` contract and must still fail closed without `new-session` or `new-window` authority.

## Helper Connection Model

Mutating commands remain per-call helper for now; long-lived helper is still deferred.

Window marking is low-frequency and task-scoped. A pooled or long-lived helper remains behind the existing separate prerequisites for state/panel/high-frequency paths: bounded request queue, cancellation propagation, crash/restart budget, capability renegotiation, and no raw output leakage.

## Preserved Boundaries

TypeScript/pi facade remains the pi extension compliance boundary.

Go may own runtime mutation only through explicit task-scoped contracts.

Still forbidden by this gate:

- `new-session`
- `new-window`
- pane creation/split/layout/resize
- wake/send-keys
- kill/clear labels
- pane labels via `set-option -p`
- pane titles via `select-pane -T`
- `refreshWindowPaneLabels` border options (`pane-border-status`, `pane-border-format`)
- state repository
- task/report/PlanRun governance
- team panel view-model
- release/package verification
- native artifact path or binary rename
- package manager native delivery or lifecycle hooks

`new-session`, `new-window`, pane creation/split/layout/resize, wake/send-keys, kill/clear labels, state/task/UI/release/package remain forbidden for this gate.

No Go source, native artifact, package version, release, or runtime facade change is made in v0.6.71.

`package.json` remains `0.6.8`.
