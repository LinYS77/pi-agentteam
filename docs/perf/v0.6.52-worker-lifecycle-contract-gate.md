# v0.6.52 Worker Lifecycle Contract Gate

Result: v0.6.52 defines the future Go worker lifecycle JSON-RPC boundary and helper connection model as a non-runtime gate.

Runtime behavior stays unchanged from v0.6.51. This slice prepares the next migration boundary without moving worker lifecycle primitives into Go.

## Contract Status

`workerLifecycle` is `design-only-not-runtime-capability`.

It is not added to active helper capabilities in this slice. The embedded helper still advertises only:

- `health`
- `profile`
- `tmuxSnapshotParse`
- `tmuxSnapshotCapture`
- `compactReadModelFingerprint`

Future JSON-RPC method: `workerLifecycle`.

The intended future request is explicitly TS-governed and compact:

```text
{
  "jsonrpc": "2.0",
  "method": "workerLifecycle",
  "params": {
    "operation": "<bounded worker lifecycle operation>",
    "governanceToken": "<opaque TS-issued call boundary marker>",
    "paneId": "<required for pane-scoped operations>",
    "teamId": "<compact identity only; no full state payload>",
    "requestId": "<TS-generated id for diagnostics and cancellation>"
  }
}
```

No Go handler for `workerLifecycle` is added in this slice.

## Future Operation Order

Worker lifecycle migration must happen in bounded phases:

- `inspectPane` and `listAgentTeamPanes` are read-only-first operations.
- `wakePane` and `syncPaneLabels` are later mutating operations.
- `createTeammatePane` is later-high-risk.
- `killPane` is last-highest-risk.

`killPane` must remain last because it is destructive and can strand visible teammate work if governance or identity mapping is wrong.

## Authority Boundary

Worker lifecycle Go authority can only be invoked by TypeScript governance/facade through an explicit adapter seam.

TypeScript/pi remains authoritative for:

- `index.ts` extension registration.
- tools and commands.
- hooks and renderers.
- TUI shell integration.
- leader-gated task governance.
- worker spawn policy and role separation.
- compact diagnostics shaping.

Go must not autonomously spawn, wake, label, create, or kill panes. No worker-spawns-worker. No hidden scheduler/autopilot/background orchestration. No peer report auto-task creation.

## Helper Connection Model

The initial helper connection model remains per-call helper invocation.

Per-call helper remains acceptable for the first worker lifecycle gate because the next worker lifecycle slices should be low-frequency and explicitly TS-governed.

Long-lived or pooled helper mode is deferred until state/panel/high-frequency paths need it.

Before any long-lived helper mode is approved, a separate gate must document and test:

- bounded request queue and backpressure policy.
- timeout and cancellation propagation per request.
- crash detection and restart budget.
- health preflight and version/capability renegotiation.
- no raw stdout/stderr/cwd/stack leakage in diagnostics.

## Preserved Boundaries

This slice does not migrate:

- worker lifecycle runtime.
- state repository.
- task/report/PlanRun governance.
- team panel view-model.
- release/package verification.
- native artifact path or binary naming.

No state repository, task/report/PlanRun, team panel view-model, release/package verification, or native artifact rename migration.

The current Go source must not add broad tmux lifecycle commands such as `display-message`, `send-keys`, `split-window`, `new-window`, or `kill-pane`. The only Go tmux command remains the v0.6.50 `tmux list-panes -a -F tmuxPaneSnapshotFormat` capture path.

No hidden TypeScript fallback is restored for `tmuxSnapshotParse` or `tmuxSnapshotCapture`.
