> Scope: Slice 3 docs/tests safety artifact only. This does not change runtime behavior, delete the TypeScript parser fallback, make Go default or required, change `package.json` version, run `npm version`, run `npm publish`, add lifecycle hooks, add package lock files, add `go.mod`/`go.sum`, add checked-in native binaries, include `kernel/` in the package, or add native packaging.

## Purpose

A future cutover-owned `tmuxSnapshotParse` parser can fail independently from tmux capture. Slice 3 records and guards the `/team` refresh safety rule: parser unavailable means an unknown/stale snapshot, not pane disappearance.

This safety rule protects light attached refresh and global refresh from destructive state updates when a parser failure produces a future fail-closed snapshot such as `ok:false`, `status:"unknown"`, `resultMarker:"stale"`, and compact `cutoverFailureKind`.

## Snapshot Classes

| Snapshot condition | Meaning | Refresh safety posture |
| --- | --- | --- |
| Capture failure | TypeScript/pi could not run or capture `tmux list-panes` | Treat as unknown/stale; do not infer pane disappearance from the missing snapshot. |
| Parser failure | TypeScript/pi captured stdout, but future cutover-owned `tmuxSnapshotParse` could not parse or validate it | Treat as unknown/stale; do not convert parser failure into `panes: []` success. |
| Successful empty snapshot | tmux capture and parser both succeeded and there are no agentteam panes in the captured universe | May be interpreted as an empty pane list only when `ok:true` is explicit and diagnostics are absent. |

Future cutover must distinguish tmux capture failure from parser failure. Both are unavailable/unknown for refresh safety, but they have different diagnostics and must not be flattened into a false successful empty pane list.

## Light Attached Refresh Safety

Ordinary attached `/team` refresh uses light reconcile intent. When the provided or captured snapshot is `ok:false`, refresh must return without destructive pane or worker mutation.

A light attached refresh with parser-unavailable `ok:false` must not:

- clear `paneId` or `windowTarget`.
- mark active workers `error`.
- write `lastWakeReason` as `pane lost` or write `lastError` as `tmux pane disappeared` solely because parser failed.
- kill panes.
- force reconcile.
- perform per-member `display-message` checks as a hidden destructive fallback.
- destructively mutate worker/member state.

The existing safety seam is `reconcileTeamPanes(team, { mode: 'light', snapshot })`: if `snapshot.ok === false`, it returns `false` and the panel data source does not write pane/member mutations.

## Global Refresh Safety

Global `/team` refresh receives one runtime snapshot and passes it to each team with light reconcile intent. If that snapshot is parser-unavailable `ok:false`, global refresh must not present it as a false successful empty pane list.

Global refresh with parser-unavailable `ok:false` must:

- keep known team `paneId` and `windowTarget` values intact.
- avoid marking active workers `error` or pane-lost solely from parser failure.
- avoid clearing pane bindings or mutating worker/member state solely from parser failure.
- avoid treating `panes: []` with `ok:false` as proof that no panes exist.
- avoid reporting parser failure as a successful empty orphan-pane universe.

Current global refresh has an explicit TypeScript/pi live tmux fallback/retry behavior for orphan pane discovery: when the shared runtime snapshot is `ok:false`, `teamPanel/dataSource.ts` calls `runtimeRepository.listAgentTeamPanes(undefined)` instead of `listAgentTeamPanesFromSnapshot(ok:false)`. That fallback is allowed only as explicit TypeScript/pi tmux authority. It must be documented and guarded as live tmux discovery/retry, not hidden parser success.

## Authority Boundaries

TypeScript/pi remains owner of:

- tmux execution and `tmux list-panes` capture.
- pane creation, pane labels, pane/window lifecycle, kill-pane behavior, light/force reconcile, and worker lifecycle.
- `/team` data source and rendering.
- state writes, repository writes, sidecar/outbox writes, task/report governance, PlanRun authority, and full-text boundaries.
- package metadata, package version, npm release flow, and native packaging decisions.

Go must not own tmux execution, pane lifecycle, worker lifecycle, `/team`, state/governance/full-text boundaries, or package/release control. A future Go-owned parser may only parse TypeScript-captured stdout after the cutover gate and runtime prerequisite signoff pass.

## Guarded Behavior

Slice 3 guards these source/runtime facts:

- `adapters/tmux/teamPanes.ts` short-circuits light reconcile on `snapshot.ok === false`.
- `teamPanel/dataSource.ts` passes `{ mode: 'light' }` for attached refresh.
- `teamPanel/dataSource.ts` passes `{ mode: 'light', snapshot }` for each team during global refresh.
- `teamPanel/dataSource.ts` uses `snapshot.ok === false ? undefined : snapshot` for global orphan pane discovery, making live tmux fallback/retry explicit and not a hidden parser success.
- `tests/suites/go-kernel-tmux-snapshot-parser.cjs` continues covering parser parity and current migration fail-open behavior; helper smokes follow `docs/perf/v0.4.19-go-helper-smoke-readiness.md` and do not approve fallback deletion.

## Stop/Go Gate

No future parser cutover or TypeScript parser fallback deletion is allowed unless refresh safety tests prove parser-unavailable snapshots are unknown/stale and non-destructive.

A future implementation that returns `ok:false` parser failure snapshots must prove:

- light attached refresh preserves pane/member state.
- global refresh preserves pane/member state.
- parser failure does not become pane disappearance.
- parser failure does not become a successful empty pane list.
- any live tmux retry/fallback remains explicit TypeScript/pi behavior.
- tmux capture failure and parser failure have distinct compact diagnostics.

Until that future implementation slice, current migration parser mode may fail open to the TypeScript parser and no runtime behavior changes are made.
