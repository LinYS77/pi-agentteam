# v0.4.20 Go Cutover Checkpoint

> Scope: final GitHub-only checkpoint review for the experimental `tmuxSnapshotParse` Go-owned cutover. This checkpoint validates Slice 1-6 coherence and reviewer evidence only. It does not make Go default or required, approve npm/native cutover, delete the TypeScript parser globally, add runtime `/team` diagnostics, change `package.json` version, run `npm version`, run `npm publish`, commit, tag, or push.

## Decision

Recommendation: GO for a GitHub-only experimental checkpoint after leader approval; STOP for npm/default/native cutover.

Accepted for this checkpoint:

- `PI_AGENTTEAM_KERNEL=go-cutover` is an explicit local/reviewer mode only.
- `tmuxSnapshotParse` is the only Go-owned cutover module.
- Source-only helper smoke evidence may be gathered by building the helper under `/tmp` with `GO111MODULE=off` and passing `PI_AGENTTEAM_KERNEL_HELPER="$helper"` explicitly.
- The TypeScript/pi control plane remains authoritative for `/team`, tmux capture/execution, worker lifecycle, state/repository writes, task/report governance, PlanRun, mailbox/report full-text boundaries, package metadata, and release operations.

Not accepted by this checkpoint:

- default Go runtime,
- required Go runtime,
- native helper packaging,
- helper build/download/package scripts,
- npm lifecycle hooks,
- package lockfiles,
- root/helper `go.mod` or `go.sum`,
- checked-in native artifacts,
- `kernel/` package inclusion,
- `npm version`,
- `npm publish`,
- TypeScript fallback deletion outside explicit `go-cutover`.

## Slice 1-6 Review Summary

- Slice 1 mode plumbing: `go-cutover` is a known explicit mode, unset/default remains `disabled`/TypeScript, and `go`/`auto`/`typescript` behavior remains migration fail-open.
- Slice 2 fail-closed parser: `tmuxSnapshotParse` failures in `go-cutover` return `ok:false`, `status:"unknown"`, `resultMarker:"stale"`, `module:"tmuxSnapshotParse"`, `capability:"tmuxSnapshotParse"`, compact `cutoverFailureKind`, empty `panes`, empty `byPaneId`, and do not invoke the TypeScript parser fallback callback.
- Slice 3 failure/no-leak coverage: covered failure classes include `missing-helper`, `disabled-helper`, `helper-unsupported-protocol`, `helper-unsupported-version`, `helper-unsupported-capability`, `helper-timeout`, `helper-spawn-error`, `helper-crash`, `helper-nonzero-exit`, `helper-empty-response`, `helper-malformed-json`, `helper-jsonrpc-error`, `helper-incompatible-response`, `helper-unsafe-response-shape`, and `previous-helper-failure`; runtime UI/data-source surfaces must not leak helper paths, stdout/stderr bodies, repo paths, mailbox/report text, raw `cutoverReason`, or full diagnostics.
- Slice 4 refresh safety: attached and global `/team` light refresh must treat cutover parser unavailable as unknown/stale rather than pane loss, avoid destructive member mutation, and avoid hidden live TypeScript orphan parsing for cutover-marked snapshots.
- Slice 5 boundary guardrails: Go helper remains parser-only/stdin-stdout; it must not execute tmux, shell out, read/write repository/state/package files, open network listeners, manage workers, own PlanRun/governance, or move full text.
- Slice 6 helper smoke: `docs/perf/v0.4.20-go-cutover-helper-smoke.md` defines `/tmp` source-only helper build, direct JSON-RPC `tmuxSnapshotParse` smoke, explicit `PI_AGENTTEAM_KERNEL=go-cutover PI_AGENTTEAM_KERNEL_HELPER="$helper"` validation, missing-helper fail-closed validation, package/native sanity, and cleanup.

## Runtime Contract Snapshot

`go-cutover` is intentionally narrower than the helper capability list:

- `AGENTTEAM_KERNEL_CUTOVER_MODULE` remains `tmuxSnapshotParse` only.
- `compactReadModelFingerprint` remains TypeScript fallback / non-cutover under `go-cutover` and must not call the helper.
- `go` and `auto` remain migration modes: helper problems may fall open to TypeScript fallback with migration `fallbackKind`/`fallbackReason`.
- `go-cutover` uses separate cutover diagnostics: helper/parser problems do not increment migration `fallbacks` and do not expose migration `fallbackKind`/`fallbackReason`.
- Generic non-cutover `ok:false` snapshots keep prior orphan fallback behavior by passing `undefined` to explicit orphan discovery; only cutover-marked parser-unavailable snapshots suppress hidden live orphan fallback.

## Package/Native Checkpoint

Required evidence before accepting the GitHub-only checkpoint:

- `package.json` version remains `0.6.8`.
- `package.json#files` excludes `kernel/`.
- no package scripts build, download, package, version, or publish the helper.
- no npm lifecycle hooks exist: `preinstall`, `install`, `postinstall`, `prepare`, `prepublish`, `prepublishOnly`, `publish`, or `postpublish`.
- no `package-lock.json` or `npm-shrinkwrap.json` exists.
- no root or helper `go.mod` exists.
- no root or helper `go.sum` exists.
- no checked-in `.exe`, `.dll`, `.so`, `.dylib`, or native helper artifact exists outside ignored dependency/build directories.
- source-only helper smoke cleans up `/tmp/agentteam-v0420-kernel.<suffix>` and leaves no `/tmp/agentteam-v0420-*` helper/build artifacts.

## Remaining Risks

- This checkpoint still relies on explicit local helper path availability; it is not a shipped helper availability model.
- GitHub-only experimental cutover evidence is acceptable for review, but npm/default/native release requires a separate native packaging/runtime prerequisite signoff.
- Runtime UI intentionally remains quiet; operators must rely on compact internal metadata/tests rather than new `/team` diagnostic rendering.
- Any future expansion beyond `tmuxSnapshotParse` must create its own cutover module gate, refresh safety analysis, no-leak tests, package/native policy, and leader-approved checkpoint.
