# v0.6.78 Go Pane Label Clearing Cutover

Result: v0.6.78 cuts over private `tmux/labels.ts clearPaneLabel(paneId, signal)` from direct TypeScript pane label/title clearing tmux calls to Go-backed `workerLifecycle.clearPaneLabel`.

This implements the runtime migration authorized by the v0.6.77 gate. It is a narrow mutating tmux cutover only for per-pane agentteam label/title clearing.

## Runtime Change

`clearPaneLabel(paneId, signal)` now delegates to `createAgentTeamKernelAdapter().clearPaneLabelAsync(paneId, signal)`.

The direct TypeScript `runTmuxNoThrowAsync(['set-option', '-up'...])` fallback is removed for the same pane label clearing behavior.

The direct TypeScript `runTmuxNoThrowAsync(['select-pane', '-t', paneId, '-T', '']...)` fallback is removed for the same pane title clearing behavior.

The private helper remains no-throw `Promise<void>` at the TypeScript boundary.

`clearPaneLabelsForTeam(...)` remains TypeScript-owned orchestration and still calls private `clearPaneLabel(...)` for each member with `member.paneId`, collects affected window targets, and then calls `refreshWindowPaneLabels(target, signal)` for each target.

## Authorized Go Command Surface

Go `workerLifecycle.clearPaneLabel` may execute only these two non-destructive pane-level tmux commands:

- `tmux set-option -up -t <paneId> @agentteam-name`
- `tmux select-pane -t <paneId> -T ''`

No other Go mutating tmux commands are introduced by this slice.

The Go implementation validates pane id compactly as a `%123`-style pane id before invoking tmux.

No label argument exists for this operation. The helper does not introduce label data and does not expose raw stdout/stderr/helper output through diagnostics.

## Compact Diagnostics and No-Throw Behavior

`clearPaneLabel(paneId, signal)` remains no-throw `Promise<void>`.

- success resolves `void`.
- helper failure resolves `void` with compact internal diagnostics.
- invalid pane id resolves `void` with compact internal diagnostics.
- abort resolves `void` with compact internal diagnostics.
- tmux unavailable resolves `void` with compact internal diagnostics.
- tmux command timeout/failure resolves `void` with compact internal diagnostics.
- raw stdout/stderr/helper output does not leak through public/private helper behavior.

The TypeScript adapter ignores helper-provided `reason`/`error` text for `clearPaneLabel` result validation and synthesizes compact diagnostics from failure kind only, so a buggy helper cannot echo raw helper/stdout/stderr content through adapter diagnostics.

## Preserved Existing Behavior

`setPaneLabel(paneId, label, signal)` remains v0.6.76 Go-backed with exactly:

- `tmux set-option -p -t <paneId> @agentteam-name <label>`
- `tmux select-pane -t <paneId> -T <label>`

The v0.6.76 opaque Unicode label argv/no raw-label leakage policy remains unchanged.

`markWindowAsAgentTeam(target, signal)` remains v0.6.72 Go-backed with exactly:

- `tmux set-option -w -t <target> automatic-rename off`
- `tmux set-option -w -t <target> allow-rename off`
- `tmux set-option -w -t <target> @agentteam-window 1`

`refreshWindowPaneLabels(target, signal)` remains v0.6.74 Go-backed with exactly:

- `tmux set-option -w -t <target> pane-border-status top`
- `tmux set-option -w -t <target> pane-border-format '#{?@agentteam-name,#{@agentteam-name},#{pane_title}}'`

Still out of scope:

- `clearPaneLabelsForTeam(...)` migration
- `new-session`
- `new-window`
- pane creation/split/layout/resize
- wake/send-keys
- kill-pane / kill-window / destructive lifecycle

- package manager native delivery or lifecycle hooks
