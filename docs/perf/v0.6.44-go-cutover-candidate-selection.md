### Why This Module

`tmuxSnapshotParse` is the best first cutover candidate because:

- It has a narrow input/output boundary: one captured stdout string plus `capturedAt`, producing compact pane snapshot data.
- It is deterministic and does not need live tmux, repository state, mailbox/report full text, provider calls, or user interaction.
- Existing parity evidence already exists in `tests/fixtures/kernel/tmux/snapshotCases.cjs` and `tests/suites/go-kernel-tmux-snapshot-parser.cjs`.
- Parser failure can be represented as unknown/stale snapshot diagnostics instead of mutating state or claiming pane loss.
- The TypeScript/pi facade can keep all product authority while delegating only parser computation after a later gate.

### Why Not A Whole Go Rewrite

A whole-product rewrite is explicitly out of scope. AgentTeam remains a TypeScript/Node pi npm extension with visible tmux teammates, leader-gated tasks, directed mailbox boundaries, durable TaskReports, `/team` TUI, package metadata, and release governance owned by TypeScript/pi.

The current risks remain seam and governance risks: state/read-model consistency, tmux capture/lifecycle, panel refresh, task/report/PlanRun authority, mailbox full-text boundaries, package/release policy, and operator evidence. Moving those responsibilities into Go would create a second control plane instead of solving the bounded parser cutover problem.

## TypeScript/pi Control-Plane Boundaries

TypeScript/pi remains authoritative for:

- pi extension loading, commands, tools, hooks, schemas, prompts, and role routing.
- tmux command execution and capture, including `list-panes`, pane labels, pane/window lifecycle, and worker lifecycle.
- task/message/report governance, TaskEvent/TaskMessageRef summaries, PlanRun control, and leader review decisions.
- state/repository/sidecar/outbox writes and compatibility with legacy state layouts.
- `/team` UI data loading/rendering and compact/full-text read boundaries.
- package metadata, package version, package inclusion, release/tag/npm governance, and rollback coordination.

Go may only own the selected module's deterministic parser implementation after a later explicit cutover gate. Go must not own tmux execution/capture, worker lifecycle, state mutation, governance, mailbox/report full-text reads, UI rendering, package/release controls, network/listener behavior, or daemon scheduling.

## Parity And Shadow Checklist

Future implementation evidence must preserve current parser semantics:

- Empty stdout returns a successful empty snapshot.
- Trailing newline and CRLF rows normalize consistently.
- Malformed rows, rows with too few fields, and empty pane ids are skipped.
- Empty labels and empty current commands are retained.
- Duplicate pane ids preserve first-seen pane order and last-seen values.
- Extra tab fields after current command are ignored.
- Unicode, long labels, long commands, and sentinel-like compact labels are retained without full-text leakage.
- `panes` and `byPaneId` remain internally consistent.
- Successful parses return `ok:true` without parser error.
- Shadow output records only compact counters, failure kinds, and corpus case ids.

## Failure Classes And Fail-Closed Diagnostics

Future cutover failures must be fail-closed for this module:

- Missing, disabled, unsupported protocol/version/capability, timeout, spawn error, crash, nonzero exit, empty response, malformed JSON, JSON-RPC error, incompatible response, unsafe response shape, and prior helper failure all produce compact diagnostics.
- Diagnostics include module/capability, failure kind, compact reason, and redacted path/basename-only details when needed.
- Diagnostics omit raw stdout, raw stderr, stack traces, absolute paths, provider ids, tool-call ids, secrets, terminal raw logs, state archives, worker transcripts, and full mailbox/report bodies.
- Parser failure is unknown/stale snapshot state; it is not proof that panes disappeared.
- Parser failure must not clear pane bindings, kill panes, mark workers error, force reconcile, write state, or emit a successful empty snapshot.

## Release Rollback Path

After a future module cutover, rollback is release governance, not silent runtime fallback. Acceptable rollback paths are:

- Revert users to a prior known-good GitHub tag/npm version.
- Publish a corrected npm version with the Go-owned parser fixed or disabled through an explicitly reviewed release path.
- Document the failure class, affected version, operator guidance, and follow-up guard before restoring forward progress.

Rollback must not expand Go authority into tmux execution, worker lifecycle, state writes, task/report/PlanRun governance, full-text reads, UI, or package control.
