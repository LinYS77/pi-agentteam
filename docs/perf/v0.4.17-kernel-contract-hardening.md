# v0.4.17 Go Kernel Contract Hardening

> Scope: Slice 0 contract inventory and freeze line for the optional Go kernel route. This records the current v0.4.16 GitHub-only checkpoint facts from commit/tag context `1578388` / `v0.4.16`; it is documentation/reference-test hardening only.

## Freeze Line

v0.4.17 Slice 0 freezes the existing optional helper contract before any parity corpus expansion:

- `protocolVersion` stays `1`.
- `adapterVersion` and `helperVersion` stay `0.3.0-read-model-shadow`; the adapter/helper string should not be bumped unless wire shape changes, result shape changes, and tests/docs are updated together.
- Required capabilities stay `health`, `profile`, `tmuxSnapshotParse`, `tmuxSnapshotCapture`, and `compactReadModelFingerprint` after the v0.6.50 protocol evolution. The original v0.4.17 freeze line covered the pre-control-plane-expansion subset without `tmuxSnapshotCapture`.
- `businessPathsConnected` must remain `false`.
- Go remains optional, source-only, read-only, benchmark/shadow-scoped, and non-authoritative.
- TypeScript/pi remains the mandatory control plane and compatibility fallback.

This slice does not implement the v0.4.17 fixture corpus, does not change runtime behavior, and does not claim Go is faster, default, packaged, or authoritative.

Forward posture: this optional/source-only/fail-open fallback matrix is v0.4.16/v0.4.17 pre-cutover migration behavior, not the final architecture for future Go-owned modules. Future module ownership must follow `docs/decisions/0002-module-owned-go-kernel-cutover.md` and `docs/perf/v0.4.18-go-module-cutover-checklist.md`: pass a cutover gate, define and execute TypeScript runtime fallback deletion, and after cutover fail closed with compact diagnostics plus GitHub tag/npm version release rollback instead of hidden runtime fallback. The v0.4.17 facts remain unchanged: Go is optional/source-only/non-authoritative for this checkpoint, with no package/native release and no runtime `/team` diagnostics.

## JSON-RPC Facts

The helper reads newline-delimited JSON-RPC 2.0 requests from stdin and writes one JSON response per non-empty input line.

Current request shape:

```json
{ "jsonrpc": "2.0", "id": "health-1", "method": "health", "params": {} }
```

Current response requirements:

- `health` returns `ok: true`, `implementation: "go"`, `protocolVersion: 1`, `helperVersion: "0.3.0-read-model-shadow"`, the required capability list, and `businessPathsConnected: false`.
- `profile` returns the health fields plus `profile.scope: "skeleton-only"`, echoed params, `stateConnected: false`, `tmuxConnected: false`, `tmuxSnapshotParseConnected: true`, `tmuxSnapshotCaptureConnected: true`, `compactReadModelFingerprintConnected: true`, `panelConnected: false`, and `taskReportPlanRunConnected: false`.
- `tmuxSnapshotParse` accepts snapshot `stdout` and `capturedAt`; it returns `capturedAt`, ordered `panes`, `byPaneId`, and `ok: true`. Empty/malformed rows and rows without pane ids are skipped; duplicate pane ids keep first-seen order with last row values.
- `tmuxSnapshotCapture` is the v0.6.50 narrow control-plane expansion capability. It executes only `tmux list-panes -a -F tmuxPaneSnapshotFormat`, reuses the Go parser, and returns either a compact successful `TmuxSnapshot` or explicit unknown/stale no-leak diagnostics.
- `compactReadModelFingerprint` accepts TypeScript-supplied compact input only; it returns `ok: true`, `projection`, `fingerprint`, `inputKind: "compact-panel-data"`, `readOnly: true`, `fullTextIncluded: false`, `stateFilesRead: false`, and `stateFilesWritten: false`.

Current helper error behavior uses JSON-RPC 2.0 error responses for parse errors (`-32700`), invalid JSON-RPC versions (`-32600`), and unknown methods (`-32601`). Current helper input scanning allows a 16 MiB line; the TypeScript adapter invokes one helper subprocess per call with compact stdin and validates the first non-empty stdout line.

| Requested mode | Helper availability | Current behavior |

After the first runtime helper failure on an adapter instance, current behavior disables helper use for that adapter and continues through TypeScript fallback without repeatedly spawning the helper.

## Compatibility Rules

Current v0.4.16 adapter compatibility is intentionally strict:

- JSON-RPC response version must be `"2.0"`; otherwise fallback kind is `helper-unsupported-protocol`.
- Helper health `protocolVersion` must equal `1`; mismatch maps to `helper-unsupported-version`.
- Helper health `helperVersion` must equal `0.3.0-read-model-shadow`; mismatch maps to `helper-unsupported-version`.
- Required capabilities must all be present.
- Unknown extra capabilities are rejected by the current adapter and map to `helper-unsupported-capability`; future additive capability policy must be deliberate and documented before tests encode it as compatible.
- `businessPathsConnected` must be `false`; `true` maps to `helper-incompatible-response`.
- Operation results must match the validated TypeScript shapes; malformed or unsafe results fall back to TypeScript.

## Slice 5 Fallback Policy

v0.4.17 Slice 5 adds `tests/suites/go-kernel-fallback-policy.cjs` to lock the operation-class fallback policy for the optional helper. The current helper-backed operations are read-only transforms over TypeScript-authorized compact inputs, so they intentionally fail open to TypeScript when the helper is unavailable, incompatible, malformed, slow, crashes, emits JSON-RPC errors, returns unsafe shapes, or fails after a prior successful health preflight. Read-only shadow/fingerprint operations fail open to TypeScript and must continue to report `readOnly:true`, `fullTextIncluded:false`, `stateFilesRead:false`, and `stateFilesWritten:false`. Tmux parser helper failures fail open to the TypeScript parser; tmux stdout capture, subprocess failures, pane lifecycle, and force/light reconcile remain TypeScript-owned and are not retried or reinterpreted by Go.

Mode policy remains explicit. `disabled` and `typescript` modes never spawn the helper, never set `fallbackKind`, and avoid scary diagnostics. `auto` with a missing helper is a quiet TypeScript fallback with no startup fallback diagnostics. `auto` with a present but incompatible helper records one compact fallback kind/reason and disables helper use for that adapter instance. `go` with a missing or incompatible helper records a compact fallback kind/reason and uses TypeScript for the current read-only/parser operations. After the first runtime helper failure on one adapter instance, later read-only/parser calls stay TypeScript-only; the helper is not spawned again and the fallback count stays stable.

Future write-side candidates fail closed by default. No TypeScript retry is allowed for a write-side helper failure; in short, no TS retry unless the operation is proven idempotent/retry-safe under the TypeScript lock and a future design adds explicit tests for exactly that retry policy. Slice 5 adds no write-side operation, no state/sidecar/cache/index authority, no repository authority, no runtime `/team` diagnostics, no task/report/PlanRun governance path, and no full-text boundary change.

Fallback diagnostics must stay compact and non-sensitive: no helper stdout/stderr bodies, full helper paths, mailbox/report full text, repository paths, state files, sidecar contents, cache/index contents, or hidden runtime state. Parity/checkpoint suites may fail when they explicitly assert Go-vs-TypeScript parity; runtime/user-facing read-only/parser paths must not fail solely because the optional helper failed.

## Non-Authority Boundaries

TypeScript/pi remains authoritative for:

- pi extension loading, tools, commands, hooks, prompts, package metadata, and npm distribution;
- state/repository reads and writes, state migrations, sidecars, caches, indexes, and legacy compatibility;
- `/team` runtime UI, panel data source authority, and user-facing diagnostics;
- tmux subprocess execution, pane creation, pane labels, pane lifecycle, light/force reconcile, and worker spawn;
- task status/owner/block/close governance, TaskReport submission/review, PlanRun transitions, leader attention, and delivery effects;
- `agentteam_receive` mailbox full-text/read boundary and `agentteam_task action=report` TaskReport full-text boundary.

Go must not become a daemon, worker, scheduler, second control plane, repository owner, full-text reader, runtime UI source, or governance actor. It may only transform TypeScript-authorized compact inputs and return validated compact outputs for optional helper-backed paths.
