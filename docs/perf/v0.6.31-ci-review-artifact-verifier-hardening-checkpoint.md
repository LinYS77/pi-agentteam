# v0.6.31 CI Review Artifact Verifier Hardening Checkpoint

## Route Decision

- Route C — strict verifier/security hardening is the v0.6.31 main route.
- Route A — hosted observation is supporting checkpoint evidence only; hosted run status is `not observed locally` unless the leader later supplies an exact commit/run id result.
- Route B — second platform row is deferred; no platform beyond `linux-x64-glibc` is supported or claimed.
- Route D — package-manager clean-install proof is deferred; no package-manager install proof is claimed.

## Implemented Evidence by Slice

### Slice 1 — expected-context verifier flags

- `scripts/lib/go-helper-artifact-verifier.cjs` accepts explicit reviewer/CI expected context for target, sourceRevision, github.sha, and github.run_id.
- `scripts/verify-go-helper-artifact.cjs` exposes `--expected-target`, `--expected-source-revision`, `--expected-github-sha`, and `--expected-github-run-id`.
- Mismatches fail closed with `context-mismatch` compact/no-leak diagnostics.
- Omitting expected-context flags still supports local reviewer usage and preserves the v0.6.30 positive path.

### Slice 2 — strict bundle surface verifier

- The verifier walks extracted artifact roots with `lstat` and rejects symlinks, non-regular files, unsafe entries, traversal, absolute paths, and Windows backslash paths.
- The generated package subtree surface is exact: helper executable, `manifest.json`, `SHA256SUMS`, `provenance.json`, `LICENSE`, `license.json`, `attestation.intoto.jsonl`, and `artifact-index.json`.
- `artifact-index.json` uses allowlisted top-level keys and allowlisted file-row keys only.
- File rows must be strict package-relative POSIX paths under `native/tmuxSnapshotParse/<helperVersion>/<target>/` and required kinds remain exactly helper, manifest, checksums, provenance, license, license-metadata, and attestation.
- `SHA256SUMS` duplicate rows, extra rows, missing rows, and malformed rows are rejected.
- Generous verifier-only ceilings cover helper bytes, metadata bytes, and total extracted bytes.

### Slice 3 — strict workflow verification

- `.github/workflows/go-helper-review-artifact.yml` verify job computes `expected_source_revision=$(git rev-parse --verify HEAD)`.
- The verify job passes strict flags for target, sourceRevision, github.sha, and github.run_id.
- `sourceRevision` from checked-out `HEAD` is not conflated with `github.sha`.
- The workflow still has a single `linux-x64-glibc` build row and a mirrored single `linux-x64-glibc` verify row.
- Workflow permissions remain `contents: read`.
- There is one `actions/upload-artifact@v4`, one `actions/download-artifact@v4`, and `retention-days: 7`.

### Slice 4 — hosted observation runbook

- `docs/perf/v0.6.31-ci-review-artifact-verifier-hardening.md` documents `workflow_dispatch` observation steps for an exact pushed commit.
- Hosted observation requires build and verify jobs green, exactly one `ubuntu-latest / linux-x64-glibc` row, strict expected-context flags, and artifact retention 7 days.
- Hosted observation remains review-only/non-availability checkpoint evidence.
- No downloaded bundles, verifier JSON, workflow summaries, generated manifests, checksums, provenance, attestations, hosted artifacts, or native binaries are checked in.
- Hosted observation is not normal-user native availability proof.
