# v0.6.83 Go Detached New-Window Gate

Result: v0.6.83 defines the detached-branch `tmux/windows.ts ensureSwarmWindow(...)` `new-window` Go cutover gate without implementing runtime mutation.

This is gate-only evidence for exactly one future candidate: replacing `runTmuxAsync(['new-window', '-t', SWARM_SESSION, '-n', SWARM_WINDOW], undefined, signal)` when `findAgentTeamWindowTarget(SWARM_SESSION, signal)` cannot find an agentteam window after detached session handling.

No Go handler, TypeScript adapter method, `tmux/windows.ts` cutover, native helper rebuild, or package/release action is made in this slice.

## Current TypeScript-Owned Behavior

`ensureSwarmWindow(preferred, signal)` remains TypeScript-owned.

Detached missing-session creation remains the v0.6.82 Go-backed cutover and is unchanged:

- `createAgentTeamKernelAdapter().createDetachedSwarmSessionAsync(SWARM_SESSION, SWARM_WINDOW, signal)`

After detached session handling, the current agentteam window discovery remains unchanged:

- `findAgentTeamWindowTarget(SWARM_SESSION, signal)`

When that lookup does not find an agentteam window, the current detached window creation command remains TypeScript-owned and unchanged:

- `runTmuxAsync(['new-window', '-t', SWARM_SESSION, '-n', SWARM_WINDOW], undefined, signal)`

The current direct `runTmuxAsync(...)` call throws on tmux command failure. This gate does not change that behavior.

After the current TypeScript-owned `new-window` call, existing lookup and marking remain unchanged:

```ts
initialTarget = await findWindowTargetByName(SWARM_SESSION, SWARM_WINDOW, signal)
if (!initialTarget) {
  throw new Error('Failed to locate agentteam tmux window after creation')
}
await markWindowAsAgentTeam(initialTarget, signal)
```

The inside-tmux branch remains TypeScript-owned and unchanged.

Post-creation pane/binding lookup remains unchanged and outside this gate:

- `firstPaneInWindow(initialTarget, signal)`
- `resolvePaneBindingAsync(leaderPaneId, signal)`
- final `markWindowAsAgentTeam(target, signal)`
- `refreshWindowPaneLabels(target, signal)`

## Authorized Future Go Command Surface

This gate authorizes only one future tmux command for a later `workerLifecycle.createDetachedSwarmWindow` cutover:

- `tmux new-window -t <SWARM_SESSION> -n <SWARM_WINDOW>`

No other Go tmux lifecycle command is authorized by this gate.

Future Go must pass the session and window names as argv values only, never shell-interpolate them.

A future helper should compactly validate `SWARM_SESSION` and `SWARM_WINDOW` before executing tmux. Future helper diagnostics should use compact failure kinds/status. Raw tmux stdout/stderr/helper output must not leak through public errors, diagnostics, task reports, logs, or tests.

In compact policy terms: raw tmux stdout/stderr/helper output must not leak.
