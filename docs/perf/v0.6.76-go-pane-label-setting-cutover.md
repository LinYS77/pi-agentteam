# v0.6.76 Go Pane Label Setting Cutover

Result: v0.6.76 cuts over private `tmux/labels.ts setPaneLabel(paneId, label, signal)` from direct TypeScript pane label/title tmux calls to Go-backed `workerLifecycle.setPaneLabel`.

This implements the runtime migration authorized by the v0.6.75 gate. It is a narrow mutating tmux cutover only for per-pane agentteam label/title setting.

## Runtime Change

`setPaneLabel(paneId, label, signal)` now delegates to `createAgentTeamKernelAdapter().setPaneLabelAsync(paneId, label, signal)`.

The direct TypeScript `runTmuxNoThrowAsync(['set-option', '-p'...])` fallback is removed for the same pane label behavior.

The direct TypeScript `runTmuxNoThrowAsync(['select-pane', '-t', paneId, '-T', label]...)` fallback is removed for the same pane title behavior.

The private helper remains no-throw `Promise<void>` at the TypeScript boundary.

`syncPaneLabelsForTeam(...)` remains TypeScript-owned orchestration and still calls private `setPaneLabel(...)` for each member with `member.paneId`.

## Authorized Go Command Surface

Go `workerLifecycle.setPaneLabel` may execute only these two non-destructive pane-level tmux commands:

- `tmux set-option -p -t <paneId> @agentteam-name <label>`
- `tmux select-pane -t <paneId> -T <label>`

No other Go mutating tmux commands are introduced by this slice.

The label is passed as an argv value only, never shell text. The label is opaque Unicode/user-visible text and may contain emoji.

The Go implementation validates pane id compactly as a `%123`-style pane id before invoking tmux.

The adapter and Go helper enforce a `4096` UTF-16-code-unit/byte-level label argument cap before invoking tmux. Invalid-label diagnostics use only compact failure kind text and do not include the label.

## No Raw Label Leakage Policy

Raw label text must not appear in diagnostics, errors, reports, logs, or validation fixtures.

Failure classes remain compact:

- `invalid-pane-id`
- `invalid-label`
- `tmux-command-failed`
- `tmux-command-timeout`
- `tmux-unavailable`
- helper failure / abort / incompatible helper result mapped through compact cutover failure kinds

The TypeScript adapter ignores helper-provided `reason`/`error` text for `setPaneLabel` result validation and synthesizes compact diagnostics from failure kind only, so a buggy helper cannot echo the raw label through adapter diagnostics.

## Public/Private Helper Behavior

`setPaneLabel(paneId, label, signal)` remains no-throw `Promise<void>`.

- success resolves `void`.
- helper failure resolves `void` with compact internal diagnostics.
- invalid pane id resolves `void` with compact internal diagnostics.
- invalid label resolves `void` with compact internal diagnostics.
- abort resolves `void` with compact internal diagnostics.
- tmux command failure resolves `void` with compact internal diagnostics.
- raw stdout/stderr/helper output does not leak through public/private helper behavior.
- raw label text does not leak through public/private helper behavior.

## Preserved Existing Behavior

`clearPaneLabel(paneId, signal)` remains TypeScript-owned and still uses direct no-throw calls:

- `tmux set-option -up -t <paneId> @agentteam-name`
- `tmux select-pane -t <paneId> -T ''`

No `set-option -up` command was added by v0.6.76 itself; v0.6.78 later added the separately gated clear-pane-label command.

`markWindowAsAgentTeam(target, signal)` remains v0.6.72 Go-backed with exactly:

- `tmux set-option -w -t <target> automatic-rename off`
- `tmux set-option -w -t <target> allow-rename off`
- `tmux set-option -w -t <target> @agentteam-window 1`

`refreshWindowPaneLabels(target, signal)` remains v0.6.74 Go-backed with exactly:

- `tmux set-option -w -t <target> pane-border-status top`
- `tmux set-option -w -t <target> pane-border-format '#{?@agentteam-name,#{@agentteam-name},#{pane_title}}'`

Still out of scope:

- `clearPaneLabel(paneId, signal)` migration
- `new-session`
- `new-window`
- pane creation/split/layout/resize
- wake/send-keys
- kill/clear labels beyond current TypeScript clear behavior
- state repository
- task/report/PlanRun governance
- team panel view-model
- release/package verification
- native artifact path or binary rename
- package manager native delivery or lifecycle hooks
