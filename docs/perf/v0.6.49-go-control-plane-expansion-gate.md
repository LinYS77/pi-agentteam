# v0.6.49 Go Control-Plane Expansion Gate

Result: v0.6.49 accepts the user-authorized architecture direction to expand Go beyond the `tmuxSnapshotParse` parser, but only as a staged control-plane migration behind the TypeScript/pi facade.

This slice does not migrate tmux capture, worker lifecycle, state, task/report/PlanRun, UI, or package/release runtime code yet. It creates the guardable architecture gate for those later slices.

## Architecture Decision

- New active ADR: `docs/decisions/0003-go-control-plane-expansion.md`.
- ADR 0003 supersedes the future-only “Go must not own control plane” boundary in ADR 0001/0002 for work after v0.6.49.
- ADR 0001/0002 remain historical evidence for the bounded helper phase and v0.6.48 parser-only cutover.
- The TypeScript/pi facade remains the public product entry: extension registration, tool/command schemas, hooks, prompts, and TUI integration stay callable from TypeScript during migration.
- Go may become the implementation core one port at a time.

## Target Direction

The target architecture is a main-package embedded Go control-plane core behind TypeScript adapters:

```text
pi extension facade
  -> TypeScript command/tool/TUI adapters
  -> embedded Go control-plane core
  -> tmux, worker lifecycle, state, task/report/PlanRun, panel view model, package verification capabilities
```

The preferred performance shape for high-frequency control-plane work is a single embedded Go binary with a stable JSON-RPC protocol and a long-lived or pooled connection. Per-call helper spawn remains acceptable only for low-frequency proofs or parser-like paths.

## Allowed Future Capabilities

Future slices may add these Go capabilities after separate gates:

- `tmuxSnapshotCapture`: execute `tmux list-panes`, parse the snapshot, and return compact pane data.
- `workerLifecycle`: create, inspect, wake, label, and kill teammate panes only when invoked by TypeScript tool/service governance.
- `stateRepository`: read/write local state through compatibility-preserving ports.
- `taskReportPlanRun`: apply task/report/PlanRun transitions while preserving leader-gated semantics.
- `teamPanelViewModel`: build compact `/team` view-model data while the TypeScript TUI shell renders it.
- `packageReleaseVerify`: verify package/release surfaces; actual npm/tag/release actions remain human-authorized.

## Preserved Product Semantics

Migration must preserve these semantics unless a later ADR explicitly changes them:

- visible teammate work remains in tmux panes.
- leader-gated task governance remains authoritative.
- non-leader `report_done` and `report_blocked` remain report-only until leader review.
- `agentteam_receive` remains the mailbox full-text/read boundary.
- `agentteam_task action=report` remains the TaskReport full-text boundary.
- workers do not spawn workers.
- peer reports do not auto-create downstream tasks.
- PlanRun remains explicit and does not gain a hidden scheduler/autopilot.
- legacy `teams/-` stays compatible and non-destructive.

## First Implementation Slice

The first implementation slice after this gate should be `tmuxSnapshotCapture`.

It is selected because it is adjacent to the already-cut-over parser and has the smallest control-plane boundary:

```text
TypeScript caller -> Go tmux list-panes execution -> Go snapshot parse -> compact snapshot result
```

Entry criteria for that slice:

- no state writes.
- no pane lifecycle mutations.
- no task/report/PlanRun mutations.
- parity with `TMUX_PANE_SNAPSHOT_FORMAT` and current snapshot shape.
- timeout and tmux-unavailable failure classes.
- compact no-leak diagnostics for stdout, stderr, cwd, stack traces, mailbox bodies, report bodies, worker transcripts, and state archives.
- rollback/default-disable behavior that does not reintroduce hidden TypeScript parser fallback ambiguity.

## Migration Order

1. v0.6.49 architecture gate and guardable migration contract.
2. Go-owned tmux snapshot capture.
3. Go-owned worker lifecycle primitives behind TypeScript tool governance.
4. Go-owned state repository behind compatibility and temp-home migration proofs.
5. Go-owned task/report/PlanRun transitions behind leader-governance characterization tests.
6. Go-owned `/team` view-model generation behind the TypeScript TUI shell.
7. Go package/release verification helpers; actual version/tag/npm/release actions remain separately authorized.

## Release And Package Guards

This slice does not authorize:

- `npm version`.
- `npm publish`.
- creating or pushing tags.
- GitHub release assets.
- second-platform support.
- signing as a release gate.
- package-manager native delivery outside the approved main-package embedded helper layout.
- `go.mod`, `go.sum`, package lockfiles, lifecycle hooks, postinstall downloads, or runtime `go build`.

`package.json` remains `0.6.8`.
