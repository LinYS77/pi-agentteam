# v0.4.18 Go Module Cutover Checklist

> Scope: planning/checklist artifact for module-level Go cutover only. This does not change runtime behavior, package versions, npm release flow, native packaging, or the TypeScript/pi control plane.

## Purpose

This checklist turns `docs/decisions/0002-module-owned-go-kernel-cutover.md` into a per-module review gate. v0.4.16/v0.4.17 optional helper and fallback behavior remains migration scaffolding; a module becomes Go-owned only after reviewers approve a narrow cutover gate, the fail-closed diagnostics contract in `docs/perf/v0.4.18-go-cutover-fail-closed-diagnostics.md`, and a concrete TypeScript runtime fallback deletion plan.

## Per-Module Template

| Field | Required Entry |
| --- | --- |
| Module/capability | Exact port/helper method and bounded capability under review. |
| TypeScript caller/runtime fallback | Current TypeScript caller, active runtime implementation, and migration fallback path before cutover. |
| Go implementation | Go helper method/package and protocol/helper-version expectations. |
| Allowed inputs | Compact inputs Go may receive after TypeScript has already enforced the control-plane boundary. |
| Forbidden inputs | State files, repository paths, sidecar/outbox writes, task/report full text, tmux lifecycle handles, package/release metadata, or other authority-bearing data Go must not receive. |
| Outputs | Deterministic compact outputs and diagnostics returned through the TypeScript port. |
| Authority boundaries | Explicit statement that TypeScript/pi remains control plane and Go owns only this module runtime after cutover. |
| Parity corpus | Fixture/test corpus proving TypeScript and Go output equivalence before cutover. |
| Benchmark/smoke commands | Focused commands reviewers run for parity, smoke, benchmark, and helper health. |
| Migration fallback policy | Pre-cutover fail-open/fallback behavior allowed only for migration safety. |
| Post-cutover fail-closed policy | Missing, disabled, incompatible, malformed, or timed-out Go must fail closed with clear diagnostics per `docs/perf/v0.4.18-go-cutover-fail-closed-diagnostics.md`. |
| Fallback deletion steps | Files/branches/config paths that remove the TypeScript runtime fallback after cutover. |
| Rollback model | GitHub tag/npm version rollback or corrected npm release; no hidden runtime fallback. |
| Owner/reviewer | Named module owner plus release/checklist reviewer. |
| Acceptance criteria | Required PASS signals and signoff needed before the module is considered Go-owned. |

## Candidate Modules

### `tmuxSnapshotParse` Candidate

- **Module/capability:** Parse tmux snapshot rows from stdout already captured by TypeScript. Candidate-specific cutover plan: `docs/perf/v0.4.18-tmux-snapshot-parse-cutover.md`.
- **TypeScript caller/runtime fallback:** `tmux/snapshot.ts` remains the migration authority before cutover; TypeScript may fail open only during parity/shadow review.
- **Go implementation:** `tmuxSnapshotParse` JSON-RPC helper method with the frozen v0.4.17 protocol/version contract.
- **Allowed inputs:** Captured stdout text and capture metadata supplied by TypeScript.
- **Forbidden inputs:** tmux subprocess handles, `send-keys`, `display-message`, `list-panes`, pane creation/kill authority, state files, task/report full text, package metadata, or release controls.
- **Outputs:** Compact parsed pane/session/window rows plus explicit diagnostics.
- **Authority boundaries:** TypeScript/pi owns tmux lifecycle, worker sessions, `/team` rendering, and user-facing control flow.
- **Parity corpus:** `tests/fixtures/kernel/tmux/snapshotCases.cjs` with `tests/suites/go-kernel-tmux-snapshot-parser.cjs` PASS.
- **Benchmark/smoke commands:** `node tests/run.cjs`, `npm run --silent bench:team-panel-tmux`, optional source-only Go helper smokes from `docs/perf/v0.4.17-kernel-release-checklist.md`.
- **Migration fallback policy:** Pre-cutover TypeScript fallback is allowed for read-only parser migration safety and benchmark/shadow evidence only.
- **Post-cutover fail-closed policy:** Missing/incompatible Go parser fails closed for this module with a clear protocol/helper diagnostic; it must not silently re-run the TypeScript parser.
- **Fallback deletion steps:** Remove TypeScript parser from runtime fallback routing, delete or retire mode branches that re-enter it after Go selection, and update tests/docs that asserted permanent fallback.
- **Rollback model:** Restore a prior GitHub tag/npm version or publish a corrected npm version; do not keep a dormant TypeScript production parser fallback.
- **Owner/reviewer:** Module owner and release reviewer must be named in the cutover PR/checklist.
- **Acceptance criteria:** Parity PASS, failure coverage PASS, fail-closed diagnostic PASS, fallback deletion reviewed, boundary scans clean, rollback text present, reviewer signoff recorded.

### `compactReadModelFingerprint` Candidate

- **Module/capability:** Build deterministic compact read-model fingerprints/projections from TypeScript-authorized compact inputs.
- **TypeScript caller/runtime fallback:** `core/readModelFingerprint.ts` remains migration authority before cutover; no runtime authority transfer until parser cutover pattern is proven.
- **Go implementation:** `compactReadModelFingerprint` JSON-RPC helper method with the frozen v0.4.17 protocol/version contract.
- **Allowed inputs:** Compact, already-authorized summaries needed for fingerprint calculation.
- **Forbidden inputs:** raw repository state files, mailbox/report full text, PlanRun authority, task governance decisions, `/team` UI authority, sidecar/outbox writes, or package/release metadata.
- **Outputs:** Compact fingerprint/projection data and explicit diagnostics; no full-text payloads.
- **Authority boundaries:** TypeScript/pi owns `/team` data-source authority, task/report governance, full-text read boundaries, and state compatibility.
- **Parity corpus:** `tests/fixtures/kernel/read-model/panelCases.cjs` with `tests/suites/go-kernel-read-model-shadow.cjs` PASS.
- **Benchmark/smoke commands:** `node tests/run.cjs`, `npm run --silent bench:state-read-model`, optional source-only Go helper smokes from `docs/perf/v0.4.17-kernel-release-checklist.md`.
- **Migration fallback policy:** Pre-cutover TypeScript fallback is allowed for read-only shadow evidence only.
- **Post-cutover fail-closed policy:** Missing/incompatible Go fingerprint runtime fails closed for this module with clear diagnostics; it must not silently recalculate through the old TypeScript runtime path.
- **Fallback deletion steps:** Remove TypeScript fingerprint runtime fallback routing only after parity/failure/boundary signoff and update tests/docs away from permanent fallback assumptions.
- **Rollback model:** Restore a prior GitHub tag/npm version or publish a corrected npm version; do not ship a hidden dual-runtime fingerprint fallback.
- **Owner/reviewer:** Module owner and release reviewer must be named in the cutover PR/checklist.
- **Acceptance criteria:** Parser cutover lessons reviewed, parity PASS, no full-text leakage, no `/team` authority movement, fail-closed PASS, fallback deletion reviewed, rollback text present, reviewer signoff recorded.

### Deferred: State/Sidecar/Outbox Writes

- **Module/capability:** State writes, repository writes, sidecar updates, outbox writes, locks, and write-side idempotency.
- **Cutover posture:** Deferred; not eligible for v0.4.18 module ownership.
- **Reason:** These paths carry write authority, compatibility risk, governance implications, and rollback/idempotency requirements that are outside this planning slice.
- **Required future work before reconsideration:** explicit write-authority design, lock protocol, idempotency model, destructive-migration guardrails, fail-closed behavior, rollback drills, and governance/full-text boundary proof.
- **Forbidden for this slice:** no Go state writes, repository writes, sidecar/outbox mutation, task/report governance movement, or runtime implementation changes.

## Cutover Gate Requirements

A module may be marked Go-owned only when all gate items are reviewed together:

- **Parity:** TypeScript and Go outputs match across the module corpus, including legacy and edge fixtures.
- **Failure coverage:** missing helper, disabled helper, protocol/version mismatch, malformed output, timeout, and unavailable toolchain cases are covered.
- **Fail-closed behavior:** post-cutover failure stops the module with an explicit compact diagnostic per `docs/perf/v0.4.18-go-cutover-fail-closed-diagnostics.md` and does not silently invoke the old TypeScript runtime fallback.
- **Fallback deletion:** the TypeScript runtime fallback deletion steps are named, reviewed, and included in the cutover change.
- **Performance/smoke:** focused benchmark/smoke commands pass; performance evidence is interpreted only after parity and boundary checks pass.
- **Boundary scans:** Go does not gain state writes, repository writes, sidecar/outbox writes, task/report governance, full-text reads, tmux lifecycle, worker lifecycle, package metadata, or release authority.
- **Rollback:** rollback instructions use GitHub tag/npm version or corrected release, not hidden runtime fallback.
- **Review signoff:** module owner and release/checklist reviewer sign off on acceptance criteria.

## Slice Constraints

- TypeScript/pi remains the control plane for commands, tools, hooks, prompts, tmux lifecycle, worker lifecycle, `/team`, governance, state compatibility, package metadata, and release flow.
- This slice does not approve a whole Go rewrite, Go daemon/control plane, Go state writes, Go governance, Go full-text boundary movement, Go tmux lifecycle, or Go package/release control.
- Do not change `package.json` version, run `npm version`, run `npm publish`, commit/tag/push, add native packaging, add `go.mod`, add `go.sum`, add package lock files, or check in native binaries.
- Do not implement Slice C/D/E/F content or runtime behavior changes as part of this checklist.
