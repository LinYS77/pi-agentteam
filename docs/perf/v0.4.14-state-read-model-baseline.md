# v0.4.14 State / Read Model Baseline

> Scope: GitHub-only checkpoint prep for the State Store / Read Model Baseline & Profiling Gate. This is a deterministic baseline artifact, not a p95 release target pass/fail claim.

## Fixture

- Leaders: 1
- Workers: 3
- Tasks: 100
- Mailbox items: 500
- Warmup iterations: 1
- Measured iterations: 5
- Runtime/tmux: stubbed; no real tmux/pi/LLM dependency
- Full-body leak guard: fixture full mailbox/report bodies include `BENCH_STATE_READ_MODEL_FULL_BODY_SENTINEL_SHOULD_NOT_LEAK`; JSON output must not include it

## Command

```bash
npm run --silent bench:state-read-model > /tmp/agentteam-v0414-state-read-model-baseline-final.json
```

Equivalent explicit command:

```bash
PI_AGENTTEAM_PROFILE=1 node tests/bench/team-read-model-baseline.cjs
```

## Output Shape

The JSON output includes:

- `fixture` sizes and `iterations` counts
- `panel.iterations`, `panel.dataLoadMs`, and `panel.readModelMs` timing percentiles
- `fsStore.byKind.lock/read/parse/write` timing percentiles and bytes
- `fsStore.bytes` totals
- `fsStore.byCategory` and `fsStore.byCallSite` breakdowns
- `tmux.commandCount`, profiled tmux command names, and stub runtime call counts
- `implementation`, `kernel`, and `fixtureProfile` metadata for future TypeScript fallback vs optional Go-kernel parity comparisons

## Notes

- The measured warm refresh path is read-only, so this run records `read` and `parse` fsStore events while `lock` and `write` buckets remain present with zero counts.
- `tmux.commandCount` is zero because the benchmark uses a stub runtime repository; stub calls are reported separately.
- Profiling is opt-in through `PI_AGENTTEAM_PROFILE=1`; ordinary user/tool output remains unpolluted when profiling is disabled.
- `AGENTTEAM_BENCH_FIXTURE=stress` can run the larger fixture profile for scalability shape; this is still baseline/profiling data, not a release target pass/fail claim.
- Future `PI_AGENTTEAM_KERNEL=go` parity runs should compare against `PI_AGENTTEAM_KERNEL=typescript` output with matching fixture/iterations and equivalent compact output; see `docs/perf/go-kernel-parity-scaffolding.md`.
- This baseline establishes a repeatable comparison point and profiling contract for later optimization work; it does not claim final v0.5 p95 release targets are achieved.
