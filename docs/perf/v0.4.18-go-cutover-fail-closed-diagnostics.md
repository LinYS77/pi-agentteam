# v0.4.18 Go Cutover Fail-Closed Diagnostics Contract

> Scope: planning/reference contract for post-cutover fail-closed diagnostics. This is docs/tests-only; it does not implement runtime diagnostics, delete TypeScript fallback paths, change package metadata, add native packaging, or publish a release.

## Purpose

v0.4.16/v0.4.17 optional-helper fallback was migration scaffolding. v0.4.18 module cutover changes the failure contract for any module that becomes Go-owned: after cutover, missing or incompatible Go is a module failure with compact diagnostics and a safe unavailable/unknown result marker, not a reason to silently re-enter the old TypeScript runtime fallback.

This contract complements `docs/decisions/0002-module-owned-go-kernel-cutover.md`, `docs/perf/v0.4.18-go-module-cutover-checklist.md`, and candidate-specific plans such as `docs/perf/v0.4.18-tmux-snapshot-parse-cutover.md`.

## Operation Classes

| Operation class | Phase | Failure posture | Notes |
| --- | --- | --- | --- |
| Migration read-only/parser helper | Pre-cutover | fail-open to TypeScript is allowed | Used only for parity corpus, shadow reporting, benchmark smokes, and migration safety while TypeScript remains authoritative. |
| Go-owned module runtime | Post-cutover | fail-closed with compact diagnostics | No silent TypeScript runtime fallback; the module returns a safe unavailable/unknown result marker and points operators to release rollback. |
| Write-side/state/governance candidates | Deferred | fail-closed by default | Not eligible for v0.4.18 cutover without later lock, idempotency, rollback, and authority design. |

Pre-cutover fail-open behavior must not be documented or tested as a permanent production architecture. Post-cutover Go-owned modules must remove hidden runtime fallback paths and treat missing/incompatible Go as an explicit module-level failure.

## Compact Diagnostic Fields

Post-cutover diagnostics must be compact, structured, and safe to show in logs, benchmark JSON, docs examples, or future debug surfaces. A module-specific contract may add fields, but the baseline fields are:

| Field | Meaning |
| --- | --- |
| `module` | Stable module name, for example `tmuxSnapshotParse`. |
| `capability` | Go helper capability or port method that failed. |
| `status` | `unavailable`, `unknown`, or module-specific fail-closed status; never a false success. |
| `cutoverFailureKind` | Stable failure class for post-cutover failures. Pre-cutover metadata may still use `fallbackKind`. |
| `reason` | Short human-readable reason with no sensitive payloads. |
| `expectedProtocolVersion` | Protocol version expected by TypeScript. |
| `actualProtocolVersion` | Compact actual version when safely known; omit when unknown. |
| `expectedHelperVersion` | Helper version or compatibility label expected by TypeScript. |
| `actualHelperVersion` | Compact actual helper version when safely known; omit when unknown. |
| `rollbackPointer` | Pointer text such as `restore prior GitHub tag/npm version or publish corrected npm version`. |
| `resultMarker` | Safe unavailable-or-unknown result marker such as `unavailable`, `unknown`, or `stale`. |
| `businessPathsConnected` | Must remain `false` unless a future explicit design changes the boundary. |

Diagnostics must let reviewers distinguish configuration/version/runtime failure from business output without exposing raw helper output or repository state.

## Stable Failure Classes

The post-cutover `cutoverFailureKind` vocabulary must cover these classes:

- `missing-helper`
- `disabled-helper`
- `helper-unsupported-protocol`
- `helper-unsupported-version`
- `helper-unsupported-capability`
- `helper-timeout`
- `helper-spawn-error`
- `helper-crash`
- `helper-nonzero-exit`
- `helper-malformed-json`
- `helper-jsonrpc-error`
- `helper-incompatible-response`
- `helper-unsafe-response-shape`
- `previous-helper-failure`

A future implementation may alias older pre-cutover names only with a documented compatibility decision. New failure classes must be added to this contract, the module checklist, and focused tests before use.

## Leak Prohibitions

Fail-closed diagnostics must not include:

- helper stdout/stderr bodies.
- full helper paths or local executable paths.
- repository paths, working-directory paths, or raw state file paths.
- mailbox/report full text or TaskReport body text.
- sidecar/cache/index contents.
- hidden runtime state or internal scheduler state.
- raw state files, raw team JSON, inbox files, outbox files, report files, or sidecar files.
- tmux pane content, shell command output beyond the already-authorized compact parser input summary, or worker prompt text.

Use stable classes, compact helper basenames if needed, and short non-sensitive reasons instead.

## Surface Policy

For v0.4.18 this contract is a docs/reference target, not a runtime UI feature.

- Bench, docs, tests, and debug-only contexts may assert or display compact diagnostics.
- Runtime `/team` must not become noisy with Go cutover diagnostics unless a later Slice explicitly designs that user-facing surface.
- `agentteam_receive` and `agentteam_task action=report` remain the full-text boundaries; diagnostics must not move those boundaries.
- Leader/task/report governance, PlanRun decisions, tmux pane lifecycle, and worker lifecycle must not depend on hidden Go diagnostics.

## Module Behavior Expectations

After a future module cutover:

- Do not silently invoke the old TypeScript runtime fallback when Go is missing, disabled, incompatible, malformed, timed out, crashed, or unsafe.
- Return or propagate a safe unavailable/unknown result marker rather than a false successful result.
- Avoid destructive state transitions from fail-closed diagnostics alone.
- Keep module-specific semantics in the module cutover plan.
- Include a rollback pointer that directs operators to restore a prior GitHub tag/npm version or publish a corrected npm version.

For `tmuxSnapshotParse`, parser failure after cutover is an unknown/stale snapshot condition, not pane loss. It must not clear pane bindings, mark workers error, kill panes, force reconcile, or emit an empty successful pane list solely because parsing failed.

Future implementation validation must include:

- module-specific parity corpus PASS before performance interpretation.
- missing/disabled/protocol mismatch/helper version mismatch/unsupported capability/timeout/spawn/crash/nonzero/malformed JSON/JSON-RPC/incompatible response/unsafe shape/previous failure coverage.
- leak checks for stdout/stderr bodies, full helper paths, repository paths, mailbox/report text, sidecar/cache/index contents, hidden state, and raw state files.
- safe unavailable/unknown result assertions for each Go-owned module.
- no silent TypeScript runtime fallback after cutover.
- release rollback text pointing to GitHub tag/npm version or corrected npm release.

## Release Rollback Pointer

Release rollback is not a runtime fallback. A bad Go-owned module is recovered by restoring a known-good GitHub tag/npm version or publishing a corrected npm version. Hidden TypeScript production fallback after cutover is forbidden because it hides failures, expands the runtime matrix, and undermines fail-closed diagnostics.

## Non-Goals

This contract does not approve or implement:

- runtime diagnostic rendering, runtime `/team` UI changes, or user-facing alert behavior.
- deletion of TypeScript fallback paths in this slice.
- package version changes, `npm version`, `npm publish`, commit, tag, push, or release publication.
- native packaging, package manager lifecycle hooks, package lock files, `go.mod`, `go.sum`, or checked-in native binaries.
- Go state writes, repository writes, sidecar/outbox writes, task/report governance, PlanRun authority, mailbox/report full-text movement, tmux lifecycle, worker lifecycle, or package/release control.
