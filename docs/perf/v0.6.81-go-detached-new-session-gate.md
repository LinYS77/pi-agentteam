# v0.6.81 Go Detached New-Session Gate

Result: v0.6.81 defines the detached-branch `tmux/windows.ts ensureSwarmWindow(...)` `new-session` Go cutover gate without implementing runtime mutation.

This is gate-only evidence for exactly one future candidate: replacing `runTmuxAsync(['new-session', '-d', '-s', SWARM_SESSION, '-n', SWARM_WINDOW], undefined, signal)` in the detached branch when `createAgentTeamKernelAdapter().sessionExistsAsync(SWARM_SESSION, signal)` reports that the swarm session is absent.

No Go handler, TypeScript adapter method, `tmux/windows.ts` cutover, native helper rebuild, or package/release action is made in this slice.

## Current TypeScript-Owned Behavior

`ensureSwarmWindow(preferred, signal)` remains TypeScript-owned.

In the detached branch, current session detection remains Go-backed and unchanged:

- `createAgentTeamKernelAdapter().sessionExistsAsync(SWARM_SESSION, signal)`

When that result does not prove the swarm session exists, the current session creation command remains TypeScript-owned and unchanged:

- `runTmuxAsync(['new-session', '-d', '-s', SWARM_SESSION, '-n', SWARM_WINDOW], undefined, signal)`

The current direct `runTmuxAsync(...)` call throws on tmux command failure. This gate does not change that behavior.

After the current TypeScript-owned `new-session` call, existing marking remains unchanged:

```ts
await markWindowAsAgentTeam(`${SWARM_SESSION}:${SWARM_WINDOW}`, signal)
```

The detached `new-window` fallback remains TypeScript-owned and unchanged:

- `runTmuxAsync(['new-window', '-t', SWARM_SESSION, '-n', SWARM_WINDOW], undefined, signal)`

Post-creation lookup remains unchanged and outside this gate:
