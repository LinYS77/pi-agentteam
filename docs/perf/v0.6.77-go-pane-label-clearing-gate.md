# v0.6.77 Go Pane Label Clearing Gate

Result: v0.6.77 defines the next narrow mutating tmux Go cutover gate without implementing runtime mutation.

This is gate-only evidence for exactly one future candidate: private `tmux/labels.ts clearPaneLabel(paneId, signal)`.

No Go operation, TypeScript facade/helper cutover, native helper rebuild, or package/release action is made in this slice.

## Current TypeScript-Owned Behavior

`clearPaneLabel(paneId, signal)` remains TypeScript-owned and currently performs exactly two no-throw tmux calls:

- `runTmuxNoThrowAsync(['set-option', '-up', '-t', paneId, '@agentteam-name'], undefined, signal)`
- `runTmuxNoThrowAsync(['select-pane', '-t', paneId, '-T', ''], undefined, signal)`

`clearPaneLabelsForTeam(...)` remains TypeScript-owned orchestration. It calls private `clearPaneLabel(member.paneId, signal)` for each member with a pane id, collects the affected window targets, and then calls the already Go-backed `refreshWindowPaneLabels(target, signal)` for each collected target.

The orchestration is not migrated by this gate.

## Authorized Future Go Command Surface

This gate authorizes only these two future pane-level tmux commands for a later `workerLifecycle.clearPaneLabel` cutover:

- `tmux set-option -up -t <paneId> @agentteam-name`
- `tmux select-pane -t <paneId> -T ''`

No other Go mutating tmux commands are authorized by this gate.

Future Go should validate the pane id compactly using the same `%123`-style policy used by v0.6.76 `setPaneLabel`.

The future Go implementation must pass all arguments as argv values, not shell text. Future helper failure, invalid pane, abort, and tmux command failure should resolve void publicly and use compact internal diagnostics. Raw stdout/stderr/helper output must not leak through public behavior.

## Future Cutover Rules

The future cutover is not implemented here. When a later task performs it:

- TypeScript/pi remains the public facade and pi extension compliance boundary.
- `clearPaneLabelsForTeam(...)` remains TypeScript-owned orchestration.
- `clearPaneLabel(paneId, signal)` remains no-throw `Promise<void>` at the helper boundary.
- helper failure, invalid pane, abort, and tmux command failure should resolve void publicly and use compact internal diagnostics.
- raw stdout/stderr/helper output must not leak through public behavior.
- after the actual cutover, `tmux/labels.ts clearPaneLabel` must not retain direct TypeScript `runTmuxNoThrowAsync(['set-option', '-up'...])` fallback for the same label-clearing behavior.
- after the actual cutover, `tmux/labels.ts clearPaneLabel` must not retain direct TypeScript `runTmuxNoThrowAsync(['select-pane', '-t', paneId, '-T', '']...)` fallback for the same label-clearing behavior.
