# v0.6.75 Go Pane Label Setting Gate

Result: v0.6.75 defines the next narrow mutating tmux Go cutover gate without implementing runtime mutation.

This is gate-only evidence for exactly one future candidate: private `tmux/labels.ts setPaneLabel(paneId, label, signal)`.

No Go operation, TypeScript facade/helper cutover, native helper rebuild, or package/release action is made in this slice.

## Current TypeScript-Owned Behavior

`setPaneLabel(paneId, label, signal)` remains TypeScript-owned and currently performs exactly two no-throw tmux calls:

- `runTmuxNoThrowAsync(['set-option', '-p', '-t', paneId, '@agentteam-name', label], undefined, signal)`
- `runTmuxNoThrowAsync(['select-pane', '-t', paneId, '-T', label], undefined, signal)`

`syncPaneLabelsForTeam(...)` remains TypeScript-owned orchestration and calls `setPaneLabel(member.paneId, ...)` for each member with a pane id.

The `label` value is intentionally opaque and user-visible. It may contain Unicode/emoji from `formatMemberPaneLabel(member)` or `formatLeaderPaneLabel(team)`.

## Authorized Future Go Command Surface

This gate authorizes only these two future pane-level tmux commands for a later `workerLifecycle.setPaneLabel` cutover:

- `tmux set-option -p -t <paneId> @agentteam-name <label>`
- `tmux select-pane -t <paneId> -T <label>`

The future Go implementation must pass `<label>` as an argv value, not shell text. It must treat the label as opaque Unicode/user-visible text and must not log or leak the raw label in diagnostics.

Future Go should validate the pane id compactly, for example `%123`-style or another explicitly approved pane target. Future Go may apply a reasonable label length cap if needed, but invalid label diagnostics must remain compact and must not include raw label content.

No other Go mutating tmux commands are authorized by this gate.

## Future Cutover Rules

The future cutover is not implemented here. When a later task performs it:

- TypeScript/pi remains the public facade and pi extension compliance boundary.
- `syncPaneLabelsForTeam(...)` remains TypeScript-owned orchestration.
- `setPaneLabel(paneId, label, signal)` remains no-throw `Promise<void>` at the helper boundary.
- helper failure, invalid pane, invalid label, abort, and tmux command failure should resolve void publicly and use compact internal diagnostics.
- raw stdout/stderr/helper output must not leak through public behavior.
- raw label text must not leak in diagnostics.
- after the actual cutover, `tmux/labels.ts setPaneLabel` must not retain direct TypeScript `runTmuxNoThrowAsync(['set-option', '-p'...])` fallback for the same label-setting behavior.
- after the actual cutover, `tmux/labels.ts setPaneLabel` must not retain direct TypeScript `runTmuxNoThrowAsync(['select-pane', '-t', paneId, '-T', label]...)` fallback for the same label-setting behavior.

## Explicitly Out of Scope

- `tmux set-option -up -t <paneId> @agentteam-name`
