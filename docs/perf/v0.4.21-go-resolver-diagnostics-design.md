# v0.4.21 Go Resolver and Diagnostics UX Design

> Scope: Slice 4 docs/tests design only. This document specifies future packaged-helper resolver and diagnostics UX behavior before implementation. It does not implement a resolver, change runtime behavior, add package metadata, add native artifacts, change package inclusion, add package scripts, run `npm version`, run `npm publish`, commit, tag, push, approve native packaging, make Go default, or delete the TypeScript parser fallback.

## Future Resolver Precedence Design

Future resolver behavior must be implemented only in a separately approved implementation slice. The intended precedence is:

1. Explicit helper path stays Model B reviewer/local path.
   - `PI_AGENTTEAM_KERNEL_HELPER` or `AGENTTEAM_GO_KERNEL_HELPER` remains a deliberate local/reviewer override.
   - explicit helper path is not normal-user availability evidence and cannot justify default fallback deletion.
2. Future packaged helper resolver is active only in separately approved preview/default-native modes.
   - packaged helper discovery must not run in default, disabled, typescript, go, auto, or current `go-cutover` unless a later implementation slice explicitly changes that mode.
   - package resolver must use Slice 2 artifact identity, manifest, platform, checksum, and version rules.
3. Default/disabled/typescript/go/auto behavior remains unchanged until separate implementation signoff.
   - default/unset remains disabled/TypeScript.
   - disabled/typescript must not spawn or resolve a helper.
   - go/auto remain migration modes until separately approved.
4. `go-cutover` remains scoped to `tmuxSnapshotParse`.
   - `compactReadModelFingerprint` remains non-cutover TypeScript fallback and must not use packaged helper discovery.
5. Runtime authority paths must not read helper environment directly except the adapter/resolver seam.
   - `/team`, teamPanel data sources, tmux adapters, runtime repositories, state repositories, worker lifecycle, PlanRun, task/report workflows, and renderers must not read `PI_AGENTTEAM_KERNEL_HELPER`, `AGENTTEAM_GO_KERNEL_HELPER`, or future packaged helper env directly.

## Future Modes and Gates

A future explicit preview mode may be named `go-packaged-preview` or another separately approved name. The name is a placeholder and is not implemented by this Slice; the later Slice 5 implementation uses `go-packaged-preview` as the explicit preview mode.

Preview-mode constraints:

- explicit opt-in only.
- resolves packaged/generated helper only after Slice 1/2/3 signoff.
- verifies platform support, manifest checksum, helper executable permission, helper health, protocol version, helper version, capability, and package/helper version compatibility before use.
- remains scoped to `tmuxSnapshotParse`.
- does not make Go default.
- does not delete TypeScript parser fallback.
- does not broaden Go helper authority.

STOP before any default/native mode:

- normal-user availability evidence is missing.
- supported platform matrix or unsupported-platform policy is unresolved.
- package/native policy has not intentionally changed in an owner slice.
- resolver tests do not cover explicit env precedence, packaged helper discovery, unsupported platform, version skew, integrity failure, permission denied, missing helper, wrong helper, and no-leak diagnostics.
- future diagnostics UX is not accepted.
- helper artifacts lack checksum/provenance/license/executable-bit validation.
- no rollback story exists for bad package metadata or bad helper artifacts.
- packaged/default cutover path can silently invoke the TypeScript parser fallback.
- runtime authority paths read helper env directly outside the adapter/resolver seam.

No silent TS parser fallback is allowed in a future packaged/default cutover path.

## Diagnostics UX Policy

Current v0.4.20/v0.4.21 runtime `/team` stays quiet. This Slice does not add runtime UI diagnostics.

Future default/native cutover may need a compact `/team` signal because a missing native helper under default cutover could otherwise look like stale pane data. That signal is allowed only after availability/package signoff and a no-leak diagnostics guard.

Safe future user-facing fields:

- module `tmuxSnapshotParse`.
- status/result marker `unknown`/`stale`.
- stable failure kind, for example `missing-helper`, `unsupported-platform`, `helper-integrity-failed`, or `helper-permission-denied` after docs/tests approval.
- short remediation text.
- supported-platform hint.
- reinstall/rollback pointer.
- compact freshness note such as `pane freshness unknown`.

Forbidden future user-facing fields:

- helper path.
- helper stdout/stderr bodies.
- repository path or cwd path.
- mailbox/report text.
- raw `cutoverReason`.
- raw state files or raw team JSON.
- sidecar/cache/index contents.
- hidden runtime state.
- worker prompts.
- stack traces.
- package internals beyond a compact reinstall/rollback pointer.
- raw manifest contents, checksums, provenance payloads, or local filesystem layout.

Recommended compact wording shape:

```text
tmuxSnapshotParse unavailable; pane freshness unknown (missing-helper). Reinstall supported native helper or roll back to a corrected release.
```

The wording is illustrative only and is not implemented in Slice 4.

## Failure Vocabulary

Existing cutover failure kinds remain compact and valid for current `go-cutover`:

- `missing-helper`
- `disabled-helper`
- `helper-unsupported-protocol`
- `helper-unsupported-version`
- `helper-unsupported-capability`
- `helper-timeout`
- `helper-spawn-error`
- `helper-crash`
- `helper-nonzero-exit`
- `helper-empty-response`
- `helper-malformed-json`
- `helper-jsonrpc-error`
- `helper-incompatible-response`
- `helper-unsafe-response-shape`
- `previous-helper-failure`

Native-specific future candidate failure kinds:

- `unsupported-platform`
- `helper-integrity-failed`
- `helper-permission-denied`

Until approved, native-specific cases may map to existing compact kinds such as `missing-helper`, `helper-spawn-error`, `helper-unsupported-version`, or `helper-incompatible-response`. Any new failure kind must be docs/tests-gated before runtime use.

## Boundary Constraints

The Go helper remains parser-only/stdin-stdout.

The helper and resolver design must not add:

- tmux execution or tmux capture.
- shell execution.
- state or repository reads/writes.
- network clients, servers, listeners, or implicit network fetches.
- worker lifecycle authority.
- pane lifecycle authority.
- PlanRun, task/report governance, assignment routing, or leader review authority.
- mailbox/report full-text access.
- runtime `/team` rendering authority outside an approved compact diagnostic surface.
- package/release authority.
- control-plane behavior outside TypeScript/pi.

TypeScript/pi remains the control plane for tmux capture, `/team`, worker lifecycle, state/repository, task/report governance, PlanRun, full-text boundaries, package metadata, and release operations.

## Validation and Readiness Design

Package/native sanity remains unchanged in Slice 4:

- `package.json` version remains `0.6.8`.
- `package.json#files` excludes `kernel/`.
- no lifecycle hooks.
- no helper build/install/download/package/version/publish scripts.
- no `optionalDependencies` or native companion package metadata yet.
- no package lockfiles.
- no root/helper `go.mod` or `go.sum`.
- no checked-in native artifacts.

Future resolver implementation must include tests for:

- explicit helper env precedence.
- packaged helper discovery only in approved preview/default-native mode.
- unsupported platform.
- package/helper version skew.
- helper integrity failure.
- helper permission denied or non-executable artifact.
- missing helper and wrong-platform helper.
- helper health/protocol/capability/version mismatch.
- no-leak `/team` diagnostics.
- no raw helper path/stdout/stderr/repo/cwd/mailbox/report/raw state/sidecar/cache/index/worker prompt/stack trace leaks.
- no hidden TypeScript parser fallback in packaged/default cutover path.
- generic non-cutover behavior remains unchanged outside future native modes.

## STOP Conditions

STOP future resolver/default-native work if any of these are true:

- resolver is implemented before Slice 1/2/3 signoff is accepted.
- package metadata changes appear without an owner package slice.
- native artifacts, lockfiles, go modules, package scripts, lifecycle hooks, or `kernel/` package inclusion appear in Slice 4.
- `go-packaged-preview` or another future preview mode is treated as implemented by this design doc rather than by the later explicit Slice 5 implementation.
- default/disabled/typescript/go/auto behavior changes without implementation signoff.
- packaged helper resolver activates outside an approved preview/default-native mode.
- `go-cutover` expands beyond `tmuxSnapshotParse`.
- `compactReadModelFingerprint` becomes cutover-owned.
- runtime authority paths read helper env directly outside adapter/resolver seam.
- future packaged/default cutover can silently fall back to TypeScript parser.
- diagnostics leak forbidden fields.
- native-specific failure vocabulary is used in runtime without docs/tests-gated approval.
- Go helper gains tmux execution/capture, state/repository writes, network/listeners, worker lifecycle, PlanRun/governance, full-text, package/release, or control-plane authority.
- `npm version`, `npm publish`, commit, tag, push, or Slice 5 starts without explicit instruction.
