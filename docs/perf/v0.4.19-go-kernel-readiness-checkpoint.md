# v0.4.19 Go Kernel Readiness Checkpoint

> Scope: Slice 5 readiness checkpoint summary only. This does not change runtime behavior, delete TypeScript fallback paths, make Go default or required, change `package.json` version, run `npm version`, run `npm publish`, commit, tag, push, add lifecycle hooks, add package lock files, add `go.mod`/`go.sum`, add checked-in native binaries, include `kernel/` in the package, or add native packaging.

## Purpose

v0.4.19 completed readiness planning for a possible future `tmuxSnapshotParse` Go-owned module cutover. It did not perform the cutover. This checkpoint ties together Slices 1-4 and records a stop/go recommendation for whether to proceed to an actual v0.4.20 cutover.

- Slice 2 tmux fail-closed readiness: `docs/perf/v0.4.19-tmux-snapshot-fail-closed-readiness.md`.

## Slice 1 Runtime Prerequisites

Readiness result: accepted for planning, not for shipped/default cutover.

- Model A is endorsed for GitHub-only readiness: source-only/manual helper path remains pre-cutover evidence only.
- Model B is documented for explicit user-provided helper path local smoke only; it is not a packaged/default release path.
- Model C native packaging matrix is deferred and out of v0.4.19 scope.

- No default Go runtime is approved by v0.4.19.

Implication: v0.4.19 can collect readiness evidence, but it cannot remove TypeScript runtime fallback paths or ship a Go-owned runtime as default.

## Slice 2 tmux Fail-Closed Readiness

Readiness result: future failure contract documented, implementation deferred.

Future `tmuxSnapshotParse` cutover must cover these failure classes before fallback deletion:

- `missing-helper`
- `disabled-helper`
- `helper-unsupported-protocol`
- `helper-unsupported-version`
- `helper-unsupported-capability`
- `helper-timeout`
- `helper-spawn-error`
- `helper-crash`
- `helper-nonzero-exit`
- `helper-empty-response`
- `helper-malformed-json`
- `helper-jsonrpc-error`
- `helper-incompatible-response`
- `helper-unsafe-response-shape`
- `previous-helper-failure`

Expected future fail-closed shape:

- `ok:false` or an equivalent unavailable/unknown marker.
- compact `cutoverFailureKind`.
- short sanitized `reason`.
- safe `status`/`resultMarker` such as `unavailable`, `unknown`, or `stale`.
- no false successful empty snapshot.

Leak prohibitions remain in force:

- no helper stdout/stderr bodies.
- no full helper paths or repository paths.
- no mailbox/report text or TaskReport body text.
- no sidecar/cache/index/raw state contents.
- no hidden runtime state.

Parity references remain:

- `tests/fixtures/kernel/tmux/snapshotCases.cjs`.
- `tests/suites/go-kernel-tmux-snapshot-parser.cjs`.

Implication: the parser failure vocabulary is ready for review, but runtime fail-closed implementation and fallback deletion are not part of v0.4.19.

## Slice 3 Refresh Safety

Readiness result: current light/global refresh safety is documented and guarded.

- Parser unavailable means unknown/stale, not pane disappearance.
- Light attached `/team` refresh with `snapshot.ok === false` must not clear `paneId` or `windowTarget`.
- Light attached `/team` refresh must not mark active workers `error` solely because parser failed.
- Light attached `/team` refresh must not write `pane lost`, `tmux pane disappeared`, kill panes, force reconcile, or destructively mutate worker/member state solely from parser failure.
- Global `/team` refresh must not present parser failure as a false successful empty pane list.
- Current global refresh has explicit TypeScript/pi live tmux fallback/retry behavior for orphan pane discovery when the shared snapshot is `ok:false`; this is documented as explicit fallback/retry, not hidden parser success.
- Future cutover must distinguish tmux capture failure from parser failure.

Implication: `/team` refresh safety is ready as a guardrail for a future cutover, but this checkpoint does not add runtime diagnostics or change refresh behavior.

## Stop/Go Recommendation

Recommendation: STOP for actual `tmuxSnapshotParse` fallback deletion in v0.4.19.

Do NOT proceed to actual `tmuxSnapshotParse` TypeScript fallback deletion yet unless all of the following happen in a future explicitly approved milestone:

- the user explicitly approves a v0.4.20 cutover attempt.
- runtime prerequisite signoff from `docs/perf/v0.4.19-go-runtime-prerequisites.md` is accepted for the target release path.
- v0.4.18/v0.4.19 module gates are accepted together: parity, fail-closed failure coverage, refresh safety, no-leak diagnostics, helper availability, rollback plan, and package/native sanity.
- implementation tests prove the future fail-closed behavior before fallback deletion.

v0.4.19 is readiness-only. It documents prerequisites, readiness contracts, safety guardrails, and reviewer commands; it does not make Go default, required, shipped, packaged, or Go-owned at runtime.
