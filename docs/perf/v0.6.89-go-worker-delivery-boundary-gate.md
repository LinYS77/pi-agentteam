# v0.6.89 Go Worker Delivery Boundary Gate

Result: v0.6.89 is gate-only evidence for the worker delivery/wake boundary.

AgentTeam worker delivery is bridge-only TypeScript-owned outbox/bridge orchestration, not legacy terminal/tmux `send-keys` delivery. This gate does not authorize Go `send-keys`, Go `wakePane`, terminal transport revival, or a runtime cutover.

No runtime source behavior changes are made by this slice.

## Current Delivery Policy

`deliveryPolicy.ts` still exposes only `export type AgentTeamDeliveryPolicyName = 'bridge-only'` for the product delivery policy surface:

- `BRIDGE_ONLY_DELIVERY_POLICY`
- `DEFAULT_DELIVERY_POLICY = BRIDGE_ONLY_DELIVERY_POLICY`
- comments stating that these helpers do not reintroduce legacy terminal/tmux delivery modes

`config.ts` rejects `deliveryMode` with `deliveryMode_unsupported` and tells users that AgentTeam delivery is bridge-only. Legacy terminal transport selection remains an error, not a supported compatibility mode.

`core/messagePolicy.ts` may classify assignment/question attention as `worker_delivery`, but that means bridge delivery, not tmux send-keys.

## Current TypeScript-Owned Outbox/Bridge Flow

Runtime delivery remains TypeScript-owned outbox/bridge orchestration:

- `app/messageApplication.ts` enqueues `kind: 'worker_delivery_requested'` for worker/recipient attention with compact idempotency key `send-worker-delivery:<team>:<recipient>:<mailboxId>:<policyIntent>` and compact delivery metadata (`messageIds`, `requestedBy`, `reason`, `wakeHint`).
- `app/taskSideEffects.ts` enqueues `kind: 'worker_delivery_requested'` for task owner nudge delivery with compact idempotency key `task-owner-nudge-delivery:<team>:<recipient>:<mailboxId>` and compact delivery metadata (`messageIds`, `requestedBy`, `reason`, `wakeHint`).
- `tools/workerSpawnService.ts` enqueues `kind: 'worker_delivery_requested'` for initial spawn delivery with compact idempotency key `spawn-initial-worker-delivery:<team>:<worker>` and compact delivery metadata (`requestedBy`, `reason`, `wakeHint`) plus the explicit initial instruction.
- `app/outbox.ts` maps `worker_delivery_requested` to `requestWorkerDelivery` for warnings/diagnostics.
- `adapters/runtime/outboxEffectHandlers.ts` invokes `requestWorkerDelivery(...)` for the durable outbox effect.

Bridge/runtime delivery request modules remain TypeScript-owned:

- `adapters/bridge/delivery.ts` builds prompts, creates bridge delivery requests, updates member status, and returns bridge/bridge-requested delivery results.
- `runtime/bridgeRequest.ts` creates or refreshes delivery requests and marks bridge work requested.
- `runtime/deliveryRequestService.ts` owns guarded lifecycle transitions, expiry, claim validation, and maintenance.
- `state/deliveryStore.ts` owns low-level delivery request persistence and normalization.
- `runtime/bridgeDeliveryPump.ts` submits through pi bridge APIs (`sendUserMessage` / `sendMessage`), not tmux.
- `app/deliveryTypes.ts` keeps delivery result methods limited to bridge/projection/leader-attention vocabulary.

There is no current direct runtime `send-keys` implementation to cut over.

## Future Terminal/Wake Policy

Future terminal/tmux wake/send-keys is NOT authorized by this gate.

If future terminal or tmux worker wake delivery is ever requested, it requires a separate explicit design gate defining all of the following before any runtime work. That means a separate explicit design gate defining the exact command surface plus the supporting policies below:

1. exact command surface, including whether any future command is `tmux send-keys -t <paneId> ...` and the exact argv shape;
2. message/worker prompt redaction and raw body leakage policy;
3. state side effects and idempotency semantics;
4. retry, rollback, and recovery semantics;
5. interaction with bridge-only delivery policy and rollback;
6. security model for opaque prompt text and untrusted message bodies;
7. tests and manual/operator evidence.

Broader Go state/task/mailbox/governance remains blocked.
