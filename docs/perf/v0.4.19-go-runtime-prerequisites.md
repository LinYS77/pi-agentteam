# v0.4.19 Go Runtime Prerequisites Decision Matrix

> Scope: Slice 1 planning/reference artifact only. This does not change runtime behavior, delete TypeScript runtime fallback paths, change `package.json` version, run `npm version`, run `npm publish`, add lifecycle hooks, add package lock files, add `go.mod`/`go.sum`, add checked-in native binaries, include `kernel/` in the package, or approve native packaging.

## Purpose

v0.4.18 defined how a bounded module could become Go-owned after parity, fail-closed diagnostics, fallback deletion planning, and release rollback review. v0.4.19 adds the missing runtime prerequisite decision: a future Go-owned shipped/default module cannot delete its TypeScript runtime fallback until reviewers sign off on how the Go helper is available to users at runtime.

This document is a stop/go gate for future cutover work, especially `tmuxSnapshotParse`. The tmux parser's fail-closed readiness details live in `docs/perf/v0.4.19-tmux-snapshot-fail-closed-readiness.md`, refresh safety for parser-unavailable snapshots lives in `docs/perf/v0.4.19-team-refresh-parser-unavailable-safety.md`, normalized source-only helper smoke commands live in `docs/perf/v0.4.19-go-helper-smoke-readiness.md`, and the final readiness checkpoint lives in `docs/perf/v0.4.19-go-kernel-readiness-checkpoint.md`. This matrix keeps GitHub-only readiness separate from packaged/default runtime availability, so source-only helper experiments do not accidentally become a shipped Go runtime requirement.

## Decision Matrix

| Model | Availability model | Allowed use before signoff | Shipped/default release posture | Cutover effect |
| --- | --- | --- | --- | --- |
| Model A | Source-only/manual helper path | Preferred v0.4.19 posture for GitHub-only readiness, docs, parity, local source smokes, and manual helper path experiments | Not a shipped/default Go runtime; no default Go runtime is approved | pre-cutover only; TypeScript runtime fallback remains and cannot be deleted |
| Model B | Explicit user-provided helper path | Experimental/local cutover smoke only when the user deliberately supplies a helper path | not a packaged/default release path and not a default Go runtime | May inform readiness, but cannot justify deleting fallback for a default release |
| Model C | Native packaging matrix | Deferred; requires a separate design for platforms, install/update, checksums, signing/provenance, support, rollback, and failure diagnostics | Required before any shipped/default Go-owned runtime cutover | Out of v0.4.19 scope; no fallback deletion until separately approved |

## Model A: Source-Only Manual Helper Path

Model A keeps the existing source-only/manual helper path as pre-cutover migration scaffolding. Developers may build or run the Go helper from source for docs, parity, benchmark, shadow, and local smoke evidence, but the npm package does not ship or require a Go-owned runtime.

Model A means:

- no shipped Go-owned runtime until runtime availability is solved.
- no default Go runtime and no hidden requirement that users compile Go to run AgentTeam.
- no TypeScript runtime fallback deletion for any shipped/default path.
- no package version change, `npm version`, `npm publish`, native packaging, lifecycle hooks, package locks, `go.mod`, `go.sum`, checked-in native binaries, or `kernel/` package inclusion.

This is the v0.4.19 recommendation for GitHub-only readiness. It lets `tmuxSnapshotParse` continue accumulating parity/failure/boundary evidence without changing product availability.

## Model B: Explicit User-Provided Helper Path

Model B may allow an explicit user-provided helper path for experimental/local cutover smoke only. It is opt-in, local, and reviewer-visible; it must not become a packaged/default release path.

Model B constraints:

- the user must deliberately provide the helper path; no lifecycle hook, download hook, package script, or implicit discovery makes Go default.
- local smoke results may prove fail-closed diagnostics and helper protocol behavior, but they do not prove runtime availability for normal npm users.
- TypeScript runtime fallback deletion remains blocked for shipped/default releases until runtime prerequisite signoff covers availability.
- diagnostics must avoid full helper paths, repository paths, helper stdout/stderr bodies, mailbox/report full text, raw state files, or sidecar/cache/index contents.

## Model C: Native Packaging Matrix

Model C is the native packaging matrix that would be required before any shipped/default Go-owned runtime cutover. It is explicitly deferred and out of v0.4.19 scope.

A future Model C design would need to define at least:

- supported OS/architecture matrix and unsupported-platform behavior.
- binary build, checksum, provenance/signing, install/update, and rollback mechanics.
- package metadata and release process changes, including whether any lifecycle hooks or downloads are acceptable.
- fail-closed diagnostics for missing, incompatible, blocked, corrupted, or unsupported binaries.
- support expectations for air-gapped, offline, source-install, and CI environments.

Until Model C or another equivalent shipped-runtime availability decision is approved, no default Go runtime is approved and TypeScript runtime fallback deletion is blocked.

## Recommendation

v0.4.19 endorses Model A for GitHub-only readiness and may document Model B as local smoke only. Model C is deferred.

Practical effect:

- keep `tmuxSnapshotParse` and other candidates in pre-cutover readiness mode.
- collect parity, failure, boundary, and diagnostics evidence without runtime behavior changes.
- do not delete TypeScript runtime fallback paths.
- do not package native helpers or make Go a default runtime dependency.

## Stop/Go Gate

Stop: no TypeScript runtime fallback deletion is allowed until runtime prerequisite signoff is recorded. The signoff must state which availability model is approved for the target release path and why normal users can run the Go-owned module without hidden local assumptions.

Go for a future shipped/default cutover requires all of the following:

- runtime prerequisite signoff covering helper availability for normal users.
- v0.4.18 cutover gate PASS for the module, including parity, failure coverage, fail-closed diagnostics, boundary scans, rollback text, and reviewer signoff.
- explicit fallback deletion targets and tests updated away from permanent fallback assumptions.
- package/native constraints reviewed and intentionally changed only in the slice that owns packaging.

Without those conditions, fallback deletion is blocked until runtime prerequisite signoff.

## Package-Neutral Constraints

Slice 1 is package-neutral. It must not:

- change `package.json` version or package files.
- run `npm version` or `npm publish`.
- add `preinstall`, `install`, `postinstall`, `prepare`, or other lifecycle hooks.
- add `package-lock.json`, `npm-shrinkwrap.json`, `go.mod`, or `go.sum`.
- check in native binaries or generated native artifacts.
- include `kernel/` in the npm package.
- add package/native release mechanics.

## Control-Plane Boundaries

TypeScript/pi remains the control plane. Go must not become:

- a daemon, scheduler, hidden worker, or second control plane.
- owner of tmux lifecycle, tmux execution, pane creation, pane labels, pane/window lifecycle, or worker lifecycle.
- owner of state writes, repository writes, sidecar/outbox writes, locks, or destructive migration.
- owner of task/report governance, PlanRun authority, assignment routing, or leader review.
- reader or mover of full-text boundaries such as `agentteam_receive` mailbox text or `agentteam_task action=report` TaskReport text.
- owner of runtime `/team` authority, package/release control, or npm metadata.

For `tmuxSnapshotParse`, Go may only parse TypeScript-captured stdout after the future module gate passes. It must not run tmux, infer pane loss from parser failure, clear state, mark workers error, or force reconcile.
