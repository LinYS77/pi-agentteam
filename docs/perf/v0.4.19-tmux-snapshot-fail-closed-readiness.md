# v0.4.19 tmuxSnapshotParse Fail-Closed Readiness

> Scope: Slice 2 docs/reference readiness only. This does not change runtime behavior, delete the TypeScript parser fallback, make Go the default runtime, change `package.json` version, run `npm version`, run `npm publish`, add lifecycle hooks, add package lock files, add `go.mod`/`go.sum`, add checked-in native binaries, include `kernel/` in the package, or add native packaging.

## Purpose

`tmuxSnapshotParse` is still a pre-cutover Go-owned module candidate. v0.4.19 Slice 2 defines the future fail-closed readiness contract for that parser without changing today's migration behavior.

The current shipped behavior remains migration fail-open: TypeScript captures tmux stdout and remains authoritative, and optional Go helper failures may fall back to the TypeScript parser for parity, shadow, benchmark, and source-only/manual helper readiness. A future cutover-owned parser mode must instead fail closed with compact diagnostics and a safe unavailable/unknown result marker; it must not silently re-run the TypeScript parser or emit a false successful empty snapshot.

## Operation Classes

| Operation class | Current/future phase | Failure posture | Runtime fallback posture |
| --- | --- | --- | --- |
| Migration parser mode | Current pre-cutover mode | fail-open to TypeScript is allowed | TypeScript parser fallback remains active for optional helper failures; this is migration scaffolding only. |
| Cutover-owned parser mode | Future mode after module gate and runtime prerequisite signoff | fail-closed with compact diagnostics | No silent TypeScript parser fallback; no false successful empty snapshot; return safe unavailable/unknown result shape. |

Current migration parser mode is allowed to collect evidence using disabled, missing, malformed, or source-only/manual helpers. That evidence must not be treated as approval to delete fallback.

Future cutover-owned parser mode may start only after runtime prerequisite signoff in `docs/perf/v0.4.19-go-runtime-prerequisites.md`; until then, fallback deletion is blocked. That signoff must cover runtime availability for the target release path, the v0.4.18 module gate, and reviewed fallback deletion targets.

## Future Cutover Failure Classes

A future `tmuxSnapshotParse` fail-closed implementation must cover these `cutoverFailureKind` classes before the parser is considered cutover-owned:

| Failure class | Trigger | Required future posture |
| --- | --- | --- |
| `missing-helper` | helper path unavailable or not installed | return unavailable/unknown; do not invoke TypeScript parser fallback after cutover. |
| `disabled-helper` | Go parser disabled by config or runtime mode | return unavailable/unknown; do not pretend Go parsed successfully. |
| `helper-unsupported-protocol` | helper protocol version is unsupported | return unavailable/unknown with expected/actual protocol when safely known. |
| `helper-unsupported-version` | helper version is unsupported | return unavailable/unknown with expected/actual helper version when safely known. |
| `helper-unsupported-capability` | helper health lacks `tmuxSnapshotParse` capability | return unavailable/unknown for this module only. |
| `helper-timeout` | helper health or parse request times out | return unavailable/unknown and preserve current pane/worker state. |
| `helper-spawn-error` | helper cannot be spawned | return unavailable/unknown without leaking full helper paths. |
| `helper-crash` | helper exits unexpectedly, crashes, or terminates by signal | return unavailable/unknown without leaking stderr bodies. |
| `helper-nonzero-exit` | helper exits nonzero for the request | return unavailable/unknown without treating stdout/stderr as diagnostics payload. |
| `helper-empty-response` | helper returns no response or an empty response | return unavailable/unknown; no false `ok:true` empty snapshot. |
| `helper-malformed-json` | helper response is not parseable JSON | return unavailable/unknown without leaking the malformed body. |
| `helper-jsonrpc-error` | helper returns a JSON-RPC error | return unavailable/unknown with compact class/reason only. |
| `helper-incompatible-response` | response is valid JSON but incompatible with the parser contract | return unavailable/unknown; do not coerce into successful panes. |
| `helper-unsafe-response-shape` | response shape is unsafe, authority-bearing, oversized, or leak-prone | return unavailable/unknown and discard unsafe details. |
| `previous-helper-failure` | prior helper health/failure state makes the parser unavailable | return unavailable/unknown until health is explicitly revalidated. |

Future implementation must add or update focused tests for every class before removing the TypeScript parser fallback. New class names must be reflected in docs/tests before runtime use.

## Expected Future Result Shape

A future cutover-owned `tmuxSnapshotParse` failure result must be compact, explicit, and safe. It may be returned directly or wrapped by a module-specific diagnostic envelope, but it must preserve these fields or equivalents:

```json
{
  "module": "tmuxSnapshotParse",
  "capability": "tmuxSnapshotParse",
  "ok": false,
  "status": "unknown",
  "resultMarker": "stale",
  "cutoverFailureKind": "missing-helper",
  "reason": "helper unavailable",
  "rollbackPointer": "restore prior GitHub tag/npm version or publish corrected npm version"
}
```

Required shape rules:

- `ok:false` or an equivalent unavailable/unknown marker is required.
- `cutoverFailureKind` must be one of the documented compact classes.
- `reason` must be short, sanitized, and non-sensitive.
- `status`/`resultMarker` must communicate `unavailable`, `unknown`, or `stale`; never false success.
- `rollbackPointer` should point to GitHub tag/npm version rollback or corrected npm release.
- The result must not emit `ok:true` with `panes: []` and `byPaneId: {}` solely because the Go parser failed.
- Parser failure is not pane loss; it must not clear pane bindings, kill panes, mark workers error, force reconcile, or destructively update worker or pane state. Refresh-specific safety is guarded in `docs/perf/v0.4.19-team-refresh-parser-unavailable-safety.md`.

## Leak Prohibitions

Future fail-closed diagnostics for `tmuxSnapshotParse` must not include:

- helper stdout/stderr bodies.
- full helper paths, local executable paths, repository paths, or working-directory paths.
- mailbox/report text, TaskReport body text, or worker prompt text.
- sidecar/cache/index/raw state contents.
- hidden runtime state, scheduler state, raw team JSON, inbox files, outbox files, report files, or sidecar files.
- tmux pane content or shell command output beyond an already-authorized compact parser input summary.

Use stable failure classes, compact helper basenames only when needed, expected/actual protocol/version fields only when safe, and short sanitized reasons.

## Stop/Go Gate

No TypeScript parser fallback deletion is allowed until all of these are true:

- runtime prerequisite signoff from `docs/perf/v0.4.19-go-runtime-prerequisites.md` covers helper availability for the target release path.
- `docs/perf/v0.4.18-tmux-snapshot-parse-cutover.md` readiness criteria pass.
- every future cutover failure class in this document has focused tests.
- compact diagnostics obey the leak prohibitions.
- package/native constraints remain explicitly reviewed by the slice that owns any packaging change.

Until then, current migration parser mode may fail open to TypeScript and no runtime behavior changes are made.
