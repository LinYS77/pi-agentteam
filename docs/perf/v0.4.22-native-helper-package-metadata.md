# v0.4.22 Native Helper Package Metadata Owner Decision

> Scope: Slice 1 metadata owner decision doc and docs guard only. This defines ownership boundaries for a future native helper package metadata schema and dry-run fixtures. It does not implement runtime resolver behavior, change `package.json`, change package version, add package metadata, add `optionalDependencies`, add lifecycle hooks, add package scripts, add helper downloads, add lockfiles, add `go.mod`/`go.sum`, add checked-in native binaries/tarballs/artifacts, include `kernel/` in package contents, run `npm version`, run `npm publish`, commit, tag, push, make Go default, approve default/native cutover, or delete the TypeScript parser fallback.

## Decision

v0.4.22 Slice 1 is GO for docs/tests-only metadata owner decision and dry-run fixture planning after leader approval.

v0.4.22 Slice 1 is STOP for all package/runtime behavior changes:

- STOP for npm version or npm publish.
- STOP for `package.json`, package version, `package.json#files`, package metadata, or native companion dependency changes.
- STOP for `optionalDependencies` or optional native deps.
- STOP for lifecycle hooks, helper install/download/build scripts, postinstall/preinstall/prepare downloads, or install-time `go build`.
- STOP for lockfiles, `go.mod`, or `go.sum`.
- STOP for checked-in native binaries, tarballs, helper executables, generated packages, manifests, or release artifacts.
- STOP for `kernel/` package inclusion.
- STOP for default/native cutover.
- STOP for TypeScript parser fallback deletion.

This slice approves only a documentable owner boundary: metadata schema and test-only dry-run fixtures can be specified; package publication and resolver defaulting remain out of scope.

## Slice 4 Manifest Compatibility and Provenance Guard

Slice 4 may add temp/generated fixture tests for future helper manifest compatibility, provenance, license, and checksum validation. This remains GitHub-only dry-run/test fixture work and does not change runtime resolver behavior.

Required manifest fields:

- `schemaVersion`.
- `package.name`.
- `package.version`.
- `helper.version`.
- `helper.protocolVersion`.
- `helper.module`.
- `helper.os`.
- `helper.arch`.
- `helper.libc`.
- `helper.filename`.
- `helper.size`.
- `helper.sha256`.
- `helper.executable`.
- `provenance.sourceRevision`.
- `provenance.generatedBy`.
- `provenance.attestation`.
- `licenses`.

Compatibility requirements:

- main package version must remain `0.6.8`.
- helper version must be `0.3.0-read-model-shadow` unless a future approved successor is documented.
- helper protocol must be `1`.
- helper module must be `tmuxSnapshotParse`.
- platform tuple must be supported.
- Linux helpers must include a Linux libc marker.
- manifest `helper.filename` must be included by the companion package `files` allowlist.
- manifest `helper.size` and `helper.sha256` must match the temp helper placeholder.
- manifest `helper.executable` must be true and the temp helper placeholder must have executable permission.
- license metadata must include the helper/package license file.
- provenance metadata must include source revision, generator, and attestation placeholders.

Rejection cases that must fail validation:

- mismatched package version.
- wrong helper version.
- wrong protocol.
- wrong module.
- unsupported platform.
- missing Linux libc.
- missing license metadata.
- missing provenance.
- missing helper.
- non-executable helper.
- size mismatch.
- checksum mismatch.
- filename/package files mismatch.

version/protocol/package/platform/checksum skew must fail closed in future preview/native resolver paths. Slice 4 only records and tests the mapping conceptually against compact fail-closed preview diagnostics vocabulary such as `missing-helper`, `helper-spawn-error`, `helper-incompatible-response`, `helper-unsupported-version`, `helper-unsupported-protocol`, and `helper-unsupported-capability`; it must not implement real resolver changes.
